## ภาพรวมระบบ (Overview)

SUT Campus Incident System ถูกพัฒนาขึ้นเพื่อปรับปรุงกระบวนการแจ้งเหตุฉุกเฉินให้มีความรวดเร็วและตรวจสอบได้ โดยมีหน้าที่หลักแยกตามบทบาทผู้ใช้งานดังนี้:

### บทบาทผู้ใช้งานหลัก (Stakeholder Roles)
*   **ผู้แจ้งเหตุ (Reporters)**: นักศึกษาหรือบุคลากรที่พบเหตุร้าย สามารถแจ้งเหตุด่วนพร้อมส่งภาพถ่ายและระบุพิกัด GPS ปัจจุบันได้ทันที
*   **เจ้าหน้าที่ศูนย์ (Dispatchers)**: ตรวจสอบรายการแจ้งเหตุผ่านแผนที่รวม คัดกรองและมอบหมายภารกิจระงับเหตุให้แก่เจ้าหน้าที่ที่เหมาะสมที่สุด
*   **เจ้าหน้าที่เผชิญเหตุภาคสนาม (Responders)**: รับมอบหมายภารกิจนำทางไปยังจุดเกิดเหตุผ่านพิกัด GPS อัปเดตขั้นตอนการระงับภัย และพิมพ์สนทนาประสานงานกับศูนย์สั่งการ

## คุณสมบัติและฟังก์ชันการทำงาน (Features)

### ฟังก์ชันสำหรับนักศึกษา/ผู้แจ้งเหตุ (User Features)
*   **ฟอร์มแจ้งเหตุฉุกเฉินความเร็วสูง**: ส่งเรื่องได้รวดเร็วผ่านการเลือกประเภทภัยหลัก (ความปลอดภัย, การแพทย์, อุบัติเหตุ, อาคารสถานที่ชำรุด, บริการทั่วไป)
*   **ระบบระบุพิกัดดาวเทียม**: ดึงพิกัดลองจิจูดและละติจูดของจุดเกิดภัยความแม่นยำสูงด้วย geolocator filter ระยะ 10 เมตร
*   **ภาพถ่ายยืนยันเหตุการณ์**: รองรับการแนบภาพถ่ายสด ณ จุดเกิดเหตุเพื่อเป็นข้อมูลให้เจ้าหน้าที่ประเมินสถานการณ์
*   **บอร์ดประกาศข่าวสาร**: ติดตามประกาศแจ้งเตือนภัย ประกาศซ่อมบำรุง หรือข้อมูลปิดเส้นทางจราจรภายในวิทยาเขต

### ฟังก์ชันสำหรับศูนย์สั่งการ (Dispatcher Features)
*   **แผงควบคุมอัจฉริยะ (Responsive Dashboard)**: มุมมองหน้าต่างแบบปรับขยายได้อัตโนมัติที่แสดงแผนที่รวม พิกัดเหตุการณ์ และข้อมูลสถิติ
*   **ระบบแนะนำเจ้าหน้าที่ระงับภัย**: จัดอันดับเจ้าหน้าที่ภาคสนามที่มีความเหมาะสมสูงสุดด้วยคะแนนการจับคู่แผนก ภาระงานสะสม และระยะทางทางภูมิศาสตร์
*   **ติดตามพิกัดเจ้าหน้าที่แบบเรียลไทม์**: แสดงตำแหน่งล่าสุดของทีมระงับเหตุบนแผนที่รวมของศูนย์สั่งการ
*   **ประวัติ**: ตรวจสอบรายการเหตุที่คลี่คลายแล้วย้อนหลัง พร้อมสรุปกราฟสถิติรายเดือน

### ฟังก์ชันสำหรับเจ้าหน้าที่เผชิญเหตุ (Responder Features)
*   **แผงเหตุส่วนตัว (Mobile Dashboard)**: จัดเรียงรายการเหตุที่ได้รับมอบหมายและลำดับความสำคัญในมุมมองที่ใช้งานง่าย
*   **ขั้นตอนการเปลี่ยนสถานะระงับ**: ปรับปรุงความคืบหน้าการทำงานอย่างเป็นลำดับขั้นตอน (`ACCEPTED` $\rightarrow$ `EN_ROUTE` $\rightarrow$ `ARRIVED` $\rightarrow$ `RESOLVED`)
*   **การนำทางพิกัดดาวเทียม**: แสดงตำแหน่งที่เกิดเหตุและระยะห่างจากพิกัดปัจจุบันของเจ้าหน้าที่
*   **ระบบคุยแชทสองช่องทาง**: ประสานงานข้อมูลผ่านแชทกลุ่มรายเหตุฉุกเฉิน หรือพิมพ์คุยส่วนตัว (DM) กับศูนย์สั่งการ

## การวิเคราะห์สถาปัตยกรรมทางเทคนิค (Architecture / System Design)

แอปพลิเคชันนี้ได้รับการพัฒนาขึ้นตามหลักการของ **Clean Architecture** ร่วมกับการจัดแบ่งกลุ่มโฟลเดอร์ตามฟีเจอร์การใช้งาน (**Feature-First Layering**):

```
        ┌────────────────────────────────────────────────────────┐
        │                 ชั้นนำเสนอข้อมูล (PRESENTATION)         │
        │  (หน้าจอ UI, วิดเจ็ต, แผนผังแบบตอบสนอง, ตัวสลับเลย์เอาต์)    │
        └───────────────────────────┬────────────────────────────┘
                                    │ คอยสังเกตการณ์ (Watches)
                                    ▼
        ┌────────────────────────────────────────────────────────┐
        │                  ชั้นตรรกะระบบ (DOMAIN)                │
        │      (กฎความถูกต้องทางธุรกิจ, กฎการเปลี่ยนสถานะเหตุการณ์)     │
        └───────────────────────────┬────────────────────────────┘
                                    │ สั่งทำงาน (Invokes)
                                    ▼
        ┌────────────────────────────────────────────────────────┐
        │                  ชั้นจัดการข้อมูล (DATA)                 │
        │  (ตัวประสาน Firebase API, เซอร์วิสไอโซเลต, ออบเจกต์สัญญา) │
        └────────────────────────────────────────────────┘
```

*   **Presentation Layer**: ตรวจจับความเคลื่อนไหวจาก Riverpod Providers โดยตรง และใช้ `LayoutBuilder` ในการปรับรูปแบบการจัดวางเมนูให้สอดรับกับขนาดหน้าจอมือถือและเดสก์ท็อปโดยอัตโนมัติ
*   **Domain Layer**: ทำหน้าที่คัดกรองและควบคุมสิทธิ์ทางธุรกิจ (เช่น กฎความปลอดภัยในไฟล์ `responder_logic.dart`) เพื่อตรวจรับรองลำดับขั้นการเปลี่ยนสเตตเหตุฉุกเฉิน
*   **Data Layer**: จัดการการเชื่อมโยงระบบภายนอกและ API ผ่านคลาส Repository ต่าง ๆ (`AuthRepository`, `IncidentRepository`, `ChatRepository`, `DirectChatRepository`) เพื่อดึง แปลง และเขียนเอกสารไปยังเซิร์ฟเวอร์คลาวด์

ศึกษาข้อมูลสถาปัตยกรรมเพิ่มเติมได้ที่ [คู่มือสถาปัตยกรรม](./docs/architecture.md)

## แพ็กเกจที่สำคัญในระบบ (Dependencies)

ไลบรารีภายนอกที่โปรเจกต์นี้พึ่งพิงหลักในการทำงาน:
*   **`flutter_riverpod`**: ควบคุมการเปลี่ยนแปลงค่าสเตตแอปพลิเคชันอย่างปลอดภัย แยกชุดคำสั่ง UI ออกจากตรรกะระบบ
*   **`cloud_firestore`**: เชื่อมต่อ เรียงลำดับ และรับส่งข้อมูลการแจ้งภัยและแชทกลุ่มแบบเรียลไทม์
*   **`firebase_auth`**: จัดเก็บเซสชันล็อกอินและรับผิดชอบการดักจับค่าอีเมลเสมือนเพื่อล็อกอินเข้าระบบ
*   **`firebase_storage`**: รับผิดชอบขั้นตอนการโอนและรับที่อยู่ไฟล์รูปภาพเกิดเหตุขึ้นคลาวด์
*   **`firebase_messaging`**: จัดการส่งรับข้อความแจ้งเตือนป๊อปอัปบนระบบปฏิบัติการ native มือถือ
*   **`cloud_functions`**: เชื่อมประสานงานแอปพลิเคชันไคลเอนต์เข้ากับบริการเซิร์ฟเวอร์เลสหลังบ้าน
*   **`google_maps_flutter`**: รันและแสดงผลแผนที่เวกเตอร์ความแม่นยำสูงบนเครื่องมือถือ Android/iOS
*   **`flutter_map`**: วาดแผนที่พิกเซลน้ำหนักเบาบนเบราว์เซอร์ PWA เพื่อป้องกันต้นทุนค่าเรียกใช้ API
*   **`geolocator`**: ค้นหาพิกัดภูมิศาสตร์ปัจจุบันของผู้ใช้ผ่านเสาสัญญาณดาวเทียม GPS
*   **`audioplayers`**: ควบคุมการถอดและเล่นไฟล์เสียงเตือนภัยไซเรนความถี่สูงบนอุปกรณ์ native
*   **`fl_chart`**: แสดงผลแผนผัง กราฟแท่งและกราฟสถิติคดีเหตุเกิดร้ายสะสมภายในขอบเขตมหาวิทยาลัย

### 1. เครื่องมือที่จำเป็นในระบบ (Prerequisites)
ก่อนเริ่มดำเนินการตรวจสอบซอร์สโค้ด กรุณาลงโปรแกรมดังต่อไปนี้ในเครื่องคอมพิวเตอร์:
*   [Flutter SDK (เวอร์ชัน v3.10.0 ขึ้นไป)](https://docs.flutter.dev/get-started/install)
*   [Node.js (เวอร์ชัน v20.x ขึ้นไป)](https://nodejs.org)
*   [Firebase CLI](https://firebase.google.com/docs/cli) (สำหรับการตั้งค่าและการเข้าถึงคลาวด์หลังบ้าน)
*   บัญชีรายชื่อโปรเจกต์ทดสอบบนบริการ Firebase คอนโซล

### 2. ขั้นตอนการรันแอปพลิเคชันหลัก (Client Setup)
เปิดเทอร์มินัลหรือคอมมานด์พร้อมต์ที่โฟลเดอร์รากของโครงการนี้ จากนั้นป้อนคำสั่งต่อไปนี้เพื่อโหลดแพ็กเกจของระบบ:
```powershell
flutter pub get
```

หลังจากดาวน์โหลดเสร็จสิ้น สามารถรันแอปพลิเคชันเพื่อวิเคราะห์ระบบได้ตามอุปกรณ์ปลายทางดังนี้:

```powershell
# คอมไพล์และเปิดระบบบนสภาพแวดล้อม Windows Desktop
flutter run -d windows

# คอมไพล์และเปิดระบบบนเบราว์เซอร์ Google Chrome (Web PWA)
flutter run -d chrome

# คอมไพล์และเปิดระบบบนเครื่องจำลองหรือโทรศัพท์มือถือที่เชื่อมต่ออยู่
flutter run
```

### 3. ขั้นตอนรันระบบคลาวด์หลังบ้าน (Cloud Functions Setup)
เข้าไปที่โฟลเดอร์หลังบ้าน `functions/` แล้วดาวน์โหลดข้อกำหนดเชื่อมต่อ Node ดังนี้:
```powershell
# เดินทางเข้าสู่โฟลเดอร์ฝั่งเซิร์ฟเวอร์เลสหลังบ้าน
cd functions

# ติดตั้งไลบรารีที่หลังบ้าน Node.js เรียกใช้
npm install
```

ขั้นตอนทดสอบและยื่นคำขอคอมไพล์ฟังก์ชันหลังบ้านขึ้นเซิร์ฟเวอร์:
```powershell
# ทดสอบคำสั่งและตรรกะระบบแบบจำลองภายในเครื่องคอมพิวเตอร์ของคุณ
firebase emulators:start

# อัปโหลดตรรกะหลังบ้านขึ้นสู่ระบบเซิร์ฟเวอร์คลาวด์ Firebase สิงคโปร์
firebase deploy --only functions
```

## ข้อมูลการตั้งค่าระบบ (Configuration)

ในการเปิดใช้งานสิทธิ์แอปพลิเคชันอย่างสมบูรณ์แบบ กรุณานำไฟล์การตั้งค่าจาก Firebase คอนโซลมาจัดวางในโปรเจกต์ดังนี้ก่อนทำการคอมไพล์:
1.  **สำหรับการคอมไพล์บน Android**: วางไฟล์ `google-services.json` ไว้ที่โฟลเดอร์ `android/app/`
2.  **สำหรับการคอมไพล์บน iOS**: วางไฟล์ `GoogleService-Info.plist` ไว้ที่โฟลเดอร์ `ios/Runner/`
3.  **สำหรับการเปิดเชื่อมต่อระบบเว็บและระบบหลัก**: สร้างไฟล์สิทธิ์ `lib/firebase_options.dart` (ผ่านคีย์สั่งการ `flutterfire configure`)

กรุณาตรวจสอบว่ามีคอลเลกชันการกำหนดค่าแอปพลิเคชันเวอร์ชัน `app_config/version` บนฐานข้อมูลหลักเพื่อเลี่ยงข้อผิดพลาดขณะระบบรันเปรียบเทียบเวอร์ชันรุ่นแอปพลิเคชัน

## ขั้นตอนและคำแนะนำการทดสอบใช้งานระบบ (Usage)

### ขั้นตอนฝั่งนักศึกษาผู้รายงานคดี (User UI Workflow)
1.  เข้าสู่แอปพลิเคชันโดยใช้รหัสนักศึกษา (เช่น `B1234567`) และป้อนรหัสผ่านที่ลงทะเบียนไว้
2.  แตะที่หมวดหมู่เหตุฉุกเฉินที่ต้องการรับความช่วยเหลือ (เช่น *เจ็บป่วยฉุกเฉิน / Medical*)
3.  กรอกรายละเอียดข้อมูล แนบภาพถ่ายปัจจุบัน ณ จุดเกิดภัย และตรวจสอบตำแหน่งหมุด GPS ให้ถูกต้อง
4.  กด **ส่งรายงานเหตุ** ซึ่งตัวแอปจะส่งพิกัดแจ้งด่วนไปปรากฏบนจอภาพสั่งการของฝ่ายรักษาความปลอดภัยโดยทันที

### ขั้นตอนฝั่งพนักงานรับแจ้งประสานงาน (Dispatcher UI Workflow)
1.  เปิดระบบสั่งการ (ตัวแอปจะปรับหน้าจอเข้าสู่อินเทอร์เฟซมอนิเตอร์โดยอัตโนมัติตามสิทธิ์ของสัญญาล็อกอิน)
2.  เลือกเหตุร้ายที่เข้ามาใหม่จากรายการแถบด้านข้างหรือแตะหมุดการแจ้งภัยสีแดงบนแผนที่หลัก
3.  พิจารณาตารางคะแนนแนะนำเจ้าหน้าที่ภาคสนาม (Smart Responder Recommendations) 
4.  แตะสั่งงานที่ปุ่ม **Assign** เพื่อยืนยันคำสั่งมอบภารกิจให้แก่เจ้าหน้าที่ทีมฉุกเฉินที่เหมาะสมที่สุด
5.  ใช้เครื่องมือแชทประสานงานข้อมูล หรือตรวจสอบการเคลื่อนที่ของเจ้าหน้าที่แบบเรียลไทม์ผ่านแผนที่มอนิเตอร์

### ขั้นตอนฝั่งเจ้าหน้าที่เผชิญภัยกู้ภัย (Responder UI Workflow)
1.  เปิดแผงภารกิจของตนเองผ่านแถบเครื่องมือ
2.  เลือกรายการเหตุที่ระบุในรายการรับงานล่าสุด
3.  แตะอัปเดตกระบวนการเข้าช่วยเหลือตามเหตุการณ์จริงเป็นลำดับความคืบหน้า (`ACCEPTED` $\rightarrow$ `EN_ROUTE` $\rightarrow$ `ARRIVED` $\rightarrow$ `RESOLVED`)
4.  เมื่อเหตุฉุกเฉินคลี่คลายเรียบร้อยแล้ว ให้กรอกแบบฟอร์มสรุปรายงานการระงับภัยเพื่อส่งคำขอปิดเคสกลับไปที่ศูนย์สั่งการ

## แบบจำลองและโครงสร้างเอกสารข้อมูล (Data Model Summary)

ข้อมูลเชิงสรุปของแผนผังการเก็บเอกสารบนระบบฐานข้อมูล NoSQL โดยรายละเอียดของชื่อตัวแปร ฟิลด์ย่อย และการตั้งกฎโครงสร้าง JSON ถูกเก็บแยกใน [เอกสารฐานข้อมูลอย่างละเอียด](./docs/database.md)

### คอลเลกชันหลัก: `users`
ทำหน้าที่ควบคุมสถิติบัญชี บทบาทหน้างาน และตำแหน่งพิกัดอัปเดตล่าสุด:
*   **ชุดฟิลด์ข้อมูลหลัก**: `uid`, `studentId`, `email`, `phone`, `role` (`user`, `responder`, `dispatcher`, `admin`), `department` (`hospital`, `rescue`, `security`), `volunteerPoints`, `lastLocation` (`lat`, `lng`, `updatedAt`)

### คอลเลกชันหลัก: `incidents`
จัดเก็บประวัติรายการเหตุด่วนและพิกัดแผนที่จุดประสบภัย:
*   **ชุดฟิลด์ข้อมูลหลัก**: `title`, `description`, `type`, `priority`, `status`, `timelineStatus`, `reporterId`, `responderId`, `latitude`, `longitude`, `imageUrls`
*   **คอลเลกชันย่อย (Subcollection)**: `messages` (เก็บข้อความแชทประสานงานรายกลุ่มเหตุฉุกเฉิน)

### คอลเลกชันหลัก: `direct_messages`
จัดเก็บแชทส่วนตัวแยกช่องทางสำหรับเจ้าหน้าที่เผชิญเหตุและวิทยุสั่งการ:
*   **ชุดฟิลด์ข้อมูลหลัก**: `participants`, `lastMessageText`, `lastMessageSenderId`, `lastMessageAt`, `lastReadBy`
*   **คอลเลกชันย่อย (Subcollection)**: `messages` (เก็บเอกสารการส่งข้อความคู่แชทสนทนา)

## การเชื่อมโยงบริการภายนอก (API & Integrations)

### คลาวด์ฟังก์ชันหลังบ้านระบบ Node.js
จัดเตรียมและลงคอนฟิกบนเซิร์ฟเวอร์หลังบ้าน **ภูมิภาคสิงคโปร์ (Singapore - `asia-southeast1`)** เพื่อป้องกันปัญหาคลาดเคลื่อนและความล่าช้าในการจัดการข้อมูลแบบเรียลไทม์:
*   `sendPushNotification`: ยิงสัญญาณแจ้งเตือนระบบป๊อปอัปความสำคัญสูงข้ามอุปกรณ์ปลายทางผ่านสิทธิ์ของคีย์ FCM Admin SDK
*   `verifyStudentByPhone`: ตรวจทานความถูกต้องเบอร์โทรและรหัสประจำตัวเพื่อความปลอดภัยก่อนเปิดให้แก้ไขข้อมูลผู้ใช้งาน
*   `resetPasswordByPhone`: รับเรื่องการตั้งประวัติปรับปรุงรหัสเข้ารหัสผ่านใหม่ในระดับแอดมินระบบรักษาความปลอดภัย

### บริการจัดการแผนที่ภูมิศาสตร์
*   **ในไคลเอนต์สมาร์ตโฟน**: ดึงความเร็วและการสัมผัสวาดแผนที่เวกเตอร์โดยเรียกชิ้นส่วนของปลั๊กอิน **Google Maps SDK**
*   **ในไคลเอนต์เว็บบราวเซอร์**: แสดงแผนที่น้ำหนักเบาด้วยพอร์ตไทล์แผนที่ฟรีของ **Leaflet OpenStreetMap (OSM)** เพื่อหลีกเลี่ยงต้นทุนค่า API แผนที่ที่แพงและซับซ้อนบนสภาพแวดล้อมเบราว์เซอร์

## ข้อจำกัดและประเด็นความปลอดภัยที่ควรทราบ (Limitations)

*   **โดเมนอีเมลมหาวิทยาลัยเสมือน**: การเข้าสู่ระบบนักศึกษาจะได้รับการดักจับเพื่อต่อท้ายเมลหลังบ้านด้วยโดเมน `@campus.local` หากมหาวิทยาลัยมีการปรับแผนหรือย้ายเปลี่ยนโครงสร้างไอทีในวิทยาเขต จะต้องปรับแก้ชุดโดเมนนี้ในสัญญาส่วนควบคุม `AuthRepository`
*   **เวลาคลาดเคลื่อนของอุปกรณ์**: ประตูสิทธิ์อ่านเขียนแชทบางฟิลด์อิงตามความตรงของระบบฐานข้อมูล Firebase (`request.time`) หากโทรศัพท์ของผู้ใช้ทั่วไปมีการเลื่อนคลาดเคลื่อนมากชั่วคราว อาจมีข้อผิดพลาดสิทธิ์ในการอ่าน
*   **ขีดจำกัดภูมิภาคคลาวด์ฟังก์ชัน**: ตรรกะ Cloud Functions จะต้องถูกปล่อยติดตั้งบนศูนย์จัดเก็บสิงคโปร์ (`asia-southeast1`) เพื่อเปิดการใช้บริการจับสัญญาณข้อมูลเชิงลึก โปรดหลีกเลี่ยงการเปลี่ยนแปลงภูมิภาคดังกล่าวระหว่างรันงานคอมไพล์

## ฟังก์ชันการขยายขอบขีดการพัฒนาในอนาคต (Future Improvements)

แนวทางการนำโปรเจกต์ไปพัฒนาต่อยอดสำหรับการทำงานจริงในวิทยาเขต:
*   **สัญญาส่งระบบตรวจสอบยืนยันเบอร์โทร SMS Gateway**: ยกระดับระบบความปลอดภัยในการกู้คืนบัญชีผู้ใช้งานผ่านรหัสยืนยันตัวตนแบบ SMS OTP
*   **ระบบคำนวณเส้นทางเดินรถและนำทางอัจฉริยะ**: จับคู่แผนที่ Leaflet ในเว็บสั่งการประสานงานเข้ากับ API ทิศทาง (เช่น OSRM) เพื่อวาดแผนที่เส้นทางจราจรที่รวดเร็วที่สุดหลบเลี่ยงปัญหาจราจรติดขัดให้แก่รถกู้ชีพ
*   **โหมดออฟไลน์ทำงานในมุมตึกอับสัญญาณ (Offline Sync)**: เปิดบันทึกเอกสารและแคชพิกัดแบบท้องถิ่น (Local SQLite/Hive) ในจังหวะที่นักศึกษาอยู่ในตึกทึบอับสัญญาณ ก่อนจะทำการอัปโหลดข้อมูลเข้า Firestore ทันทีเมื่อกลับสู่สัญญาณเน็ตปกติ
