# Team Brief: SUT Campus Incident System

ไฟล์นี้ทำไว้ให้เพื่อนอ่านเร็ว ๆ ก่อนนำเสนอ เพื่อให้ทุกคนตอบไปในทิศทางเดียวกัน

## โปรเจกต์นี้คืออะไร

SUT Campus Incident System คือแอปแจ้งเหตุและประสานงานเหตุฉุกเฉินในมหาวิทยาลัย พัฒนาด้วย Flutter และใช้ Firebase เป็นระบบหลังบ้าน จุดสำคัญคือไม่ได้มีแค่ "ส่งฟอร์มแจ้งเหตุ" แต่มี flow ครบตั้งแต่ผู้แจ้งเหตุส่งข้อมูล ศูนย์ประสานงานเห็นเหตุแบบ real-time มอบหมายเจ้าหน้าที่ และเจ้าหน้าที่อัปเดตสถานะจนปิดเคส

## คนใช้ระบบมี 4 บทบาท

1. User: นักศึกษา/บุคลากร ใช้แจ้งเหตุ ดูประวัติ แชท และรับข่าวสาร
2. Dispatcher: เจ้าหน้าที่ศูนย์ประสานงาน ดูเหตุใหม่ ตรวจสอบพิกัด และมอบหมาย responder
3. Responder: เจ้าหน้าที่ภาคสนาม รับเคส เดินทางไปจุดเกิดเหตุ และอัปเดตสถานะ
4. Admin: ผู้ดูแลระบบ ดูภาพรวม จัดการผู้ใช้/ข้อมูล และตรวจสอบระบบ

## Flow หลักที่ต้องจำ

ผู้ใช้แจ้งเหตุ -> Firestore บันทึกข้อมูล -> Dispatcher เห็นเหตุใหม่แบบ real-time -> Dispatcher มอบหมาย Responder -> Responder รับแจ้งเตือน -> Responder ไปหน้างาน -> อัปเดตสถานะ -> ปิดเคส -> ผู้ใช้ประเมิน/ตรวจสอบย้อนหลัง

## โค้ดแบ่งยังไง

- `lib/main.dart`: จุดเริ่มต้นของแอป ตั้งค่า Firebase, notification, theme, ภาษา และ route หลัก
- `lib/features/auth`: login/register และตรวจ role
- `lib/features/home`: หน้าแรกของ user
- `lib/features/incident`: ส่วนแจ้งเหตุ รายละเอียดเหตุ ประวัติ และ logic ของ incident
- `lib/features/dispatcher`: หน้าศูนย์ประสานงาน ใช้ดูเคส มอบหมายงาน ดูแผนที่
- `lib/features/responder`: หน้าของเจ้าหน้าที่ภาคสนาม รับงานและอัปเดตสถานะ
- `lib/features/chat`: แชทระหว่างผู้เกี่ยวข้องกับเคส
- `lib/features/notification`: FCM push notification และการกดแจ้งเตือนแล้วไปหน้าที่เกี่ยวข้อง
- `lib/features/map`: heatmap/แผนที่จุดเสี่ยง
- `lib/features/admin`: dashboard และจัดการผู้ใช้
- `lib/models`: โครงสร้างข้อมูลกลาง เช่น `Incident`, `UserModel`, `ChatMessage`
- `functions/index.js`: Firebase Cloud Functions เช่น ส่ง push notification และ reset password

## Tech stack ที่ควรพูด

- Flutter/Dart: ทำแอปข้ามแพลตฟอร์ม Android, iOS, Web, Windows
- Riverpod: จัดการ state และ provider
- Firebase Auth: สมัคร/เข้าสู่ระบบ
- Cloud Firestore: ฐานข้อมูล real-time
- Firebase Storage: เก็บรูป/สื่อประกอบการแจ้งเหตุ
- Firebase Cloud Messaging: ส่งแจ้งเตือน
- Cloud Functions: ทำงานฝั่ง server เช่น push notification และ reset password
- Google Maps / OpenStreetMap: แสดงพิกัดและนำทาง

## ประโยคสั้นไว้ตอบอาจารย์

"ระบบนี้เป็น emergency coordination system แบ่งเป็น 4 role โดยข้อมูล incident ถูกเก็บใน Firestore แบบ real-time แล้ว dispatcher ใช้ dashboard คัดกรองและมอบหมาย responder จากนั้น responder อัปเดต timeline ของเคสจน resolved"

## 🗺️ 1. ลำดับขั้นตอนการแชร์ความรู้ให้เพื่อนเข้าใจง่ายที่สุด (Step-by-Step Sharing)

หากคุณต้องเป็นคนพูดอธิบายให้เพื่อนในกลุ่มฟังด้วยตัวเอง ให้ทำตาม 4 ขั้นตอนนี้เพื่อไม่ให้เพื่อนสมองบวมกับเรื่องโค้ดตั้งแต่แรก:

```
[ขั้นตอนที่ 1: เล่าเรื่อง Scenario] ──> [ขั้นตอนที่ 2: อธิบาย 3 ส่วนหลัก] ──> [ขั้นตอนที่ 3: พาดูโครงสร้างไฟล์] ──> [ขั้นตอนที่ 4: ซ้อมตอบคำถาม]
```

1. **ขั้นตอนที่ 1: เล่าด้วย "ฉากทัศน์การใช้งานจริง" (Story-based Scenario)**
   * *คำอธิบาย:* "แอปเราไม่ใช่แอปแชททั่วไปนะ แต่มันคือระบบประสานงานเหตุฉุกเฉิน 3 ฝ่ายแบบเรียลไทม์ ลองนึกภาพว่ามีนักศึกษาขาหักที่ มทส. (ผู้แจ้งเหตุ) -> กดแจ้งเหตุปุ๊บ พิกัด GPS ส่งเข้าศูนย์วิทยุกลาง (Dispatcher) ทันที -> เจ้าหน้าที่ศูนย์กดมอบหมายงานให้หน่วยกู้ภัย (Responder) -> กู้ภัยได้รับแจ้งเตือนเสียงไซเรนและเปิดแผนที่นำทางไปช่วยเหลือ และคุยแชทกันได้แบบ 3 ฝ่าย"
2. **ขั้นตอนที่ 2: อธิบาย "3 เสาหลักทางเทคนิค" (High-Level Stack)**
   * *ฝั่งหน้าบ้าน (Frontend):* พัฒนาด้วย **Flutter** เขียนภาษา **Dart** ตัวเดียวรันได้ทั้งบนมือถือ (Android/iOS สำหรับคนแจ้งและกู้ภัย) และบนเว็บเบราว์เซอร์ (Web สำหรับหน้าจอศูนย์สั่งการ)
   * *ฝั่งการจัดการข้อมูล (Database & Cloud):* ใช้ **Firebase** ทั้งหมด (Auth = สมัครสมาชิก/เข้าระบบ, Firestore = ฐานข้อมูลเรียลไทม์แชทและเหตุการณ์, Storage = เก็บรูปภาพถ่ายเหตุการณ์)
   * *ฝั่งหลังบ้าน (Background Engine):* ใช้ **Firebase Cloud Functions** (Node.js) คอยตรวจจับเมื่อมีเคสใหม่ แล้วยิงแจ้งเตือนด่วน (Push Notification - FCM) ไปยังมือถือกู้ภัยทันที
3. **ขั้นตอนที่ 3: พารู้จัก "แนวคิดการจัดโครงสร้างโค้ด" (Feature-First Clean Architecture)**
   * อธิบายให้เพื่อนฟังว่า โค้ดในโปรเจกต์นี้จัดหมวดหมู่ตาม **"ฟีเจอร์" (Features)** ไม่ใช่ตามชนิดไฟล์ ทำให้หาง่ายมาก เช่น อยากดูเรื่องแชท ให้ไปที่ `lib/features/chat/`, อยากดูเรื่องแจ้งเหตุ ให้ไปที่ `lib/features/incident/`
   * ในแต่ละฟีเจอร์จะแบ่งงานเป็น 3 เลเยอร์:
     * **Presentation (UI):** หน้าจอที่ผู้ใช้เห็น (Screens & Widgets) และตัวควบคุมสเตตัส (**Riverpod Controllers**)
     * **Domain (Business Rules):** กฎการทำงาน เช่น กฎการเปลี่ยนสถานะจาก NEW -> IN_PROGRESS -> RESOLVED
     * **Data (Data Connection):** ตัวติดต่อดึงข้อมูลจาก Firebase (**Repositories**)
4. **ขั้นตอนที่ 4: แจกบทบาทสมมติ (Role Assignment) และซ้อมตอบคำถามอาจารย์**
   * (ดูรายละเอียดการแจกบทบาทและแนวคำตอบในหัวข้อถัดไป)

## 💡 3. สรุปคีย์เวิร์ดสำคัญ (Tech Keywords) ที่เพื่อนต้องจำไปพูด
*ให้เพื่อนจำคีย์เวิร์ดเหล่านี้ไปใช้พูดประกอบการตอบคำถาม จะช่วยเพิ่มความน่าเชื่อถือขึ้น 200%:*

1. **"Feature-First Clean Architecture"**
   * *ความหมาย:* การจัดโค้ดเป็นโฟลเดอร์ตามฟีเจอร์งานหลัก (เช่น `auth`, `chat`, `incident`) ทำให้ทีมงานทำงานขนานกันได้ง่าย โค้ดไม่ปนกัน และบำรุงรักษาง่าย
2. **"Riverpod (State Management)"**
   * *ความหมาย:* ตัวจัดการสถานะและข้อมูลส่วนกลางในแอป ช่วยให้เวลามีการแชทหรือข้อมูลเหตุเปลี่ยนไป หน้าจอ UI จะรีเฟรชอัปเดตสถานะใหม่ทันทีโดยไม่จำเป็นต้องโหลดหน้าจอใหม่ทั้งหมด
3. **"Adaptive Map Architecture"**
   * *ความหมาย:* ระบบแผนที่ที่เปลี่ยนเครื่องยนต์เรนเดอร์เองอัตโนมัติ บนมือถือใช้ **Google Maps SDK** เพื่อความลื่นไหล บนเว็บเบราว์เซอร์ใช้ **OpenStreetMap** เพื่อความประหยัด (ไม่มีค่าใช้จ่าย API) และเสถียร
4. **"Real-time Stream / Cloud Functions Notification"**
   * *ความหมาย:* ระบบแชทและอัปเดตพิกัดใช้ฟังก์ชัน Stream ของ Firestore ในการดึงข้อมูลสดตลอดเวลา ส่วน Push Notification ใช้ Node.js รันบน Firebase Cloud Functions ในสิงคโปร์เพื่อความรวดเร็วและปลอดภัย

## ❓ 4. แนวถาม-ตอบ ยอดฮิตสำหรับเตรียมตอบคำถามอาจารย์ (Professor Q&A Cheat Sheet)

> [!TIP]
> **วิธีการซ้อม:** ให้สมาชิกในกลุ่มลองผลัดกันถามคำถามเหล่านี้ และฝึกตอบโดยอิงตามสคริปต์ที่เตรียมไว้ให้

### คำถามที่ 1: ทำไมถึงเลือกใช้สถาปัตยกรรม Feature-First ในโปรเจกต์นี้?
* **แนวคำตอบ:**
  > "เพราะระบบของเรามีฟังก์ชันหลากหลายตามบทบาทผู้ใช้ (เช่น ผู้ประสบภัย, ศูนย์สั่งการ, และหน่วยกู้ภัย) การแบ่งสถาปัตยกรรมแบบ Feature-First ช่วยให้เราสามารถแยกความรับผิดชอบของโค้ดแต่ละโมดูลได้อย่างเด็ดขาดครับ เช่น ฟีเจอร์แชท (`chat`) จะไม่ปะปนกับฟีเจอร์ลงทะเบียนเหตุ (`incident`) ทำให้ทีมพัฒนาสามารถแบ่งงานกันทำในแต่ละโฟลเดอร์ได้อย่างอิสระ โค้ดไม่ทับไลน์กัน และเมื่อเกิดบั๊กในจุดใดจุดหนึ่ง ก็สามารถจำกัดวงการแก้ไขได้ง่ายโดยไม่กระทบกับส่วนอื่นๆ ของระบบครับ"

### คำถามที่ 2: ระบบจัดการความปลอดภัยข้อมูลผู้ใช้และการเข้าถึงข้อมูลเหตุฉุกเฉินอย่างไร? (คำถามสุดฮิต!)
* **แนวคำตอบ:**
  > "เราใช้ระบบสิทธิ์การเข้าถึงแบบ **Role-Based Access Control (RBAC)** ร่วมกับ **Firestore Security Rules** ที่ติดตั้งไว้บนระบบคลาวด์โดยตรงครับ ตัวอย่างเช่น:
  > 1. ในคอลเลกชัน `/users` สมาชิกทั่วไปจะอ่านและแก้ไขได้เฉพาะข้อมูลโปรไฟล์ตัวเองเท่านั้น ไม่สามารถเขียนฟิลด์ `role` เพื่อยกระดับสิทธิ์ให้ตัวเองกลายเป็น `admin` หรือ `dispatcher` ได้
  > 2. ในคอลเลกชัน `/incidents/{incidentId}/messages` (ห้องแชท) เราเขียนกฎล็อกไว้ว่า เฉพาะผู้แจ้งเหตุที่เป็นเจ้าของเคสนี้ และเจ้าหน้าที่กู้ภัยที่ได้รับมอบหมายเท่านั้นที่มีสิทธิ์ Read และ Write ข้อความ เพื่อความปลอดภัยสูงสุดของข้อมูลผู้ประสบภัยครับ"

### คำถามที่ 3: ระบบนี้ทำงานเป็นแบบ Real-time ได้อย่างไร? โครงสร้างฐานข้อมูลออกแบบไว้อย่างไร?
* **แนวคำตอบ:**
  > "เราเลือกใช้ **Google Cloud Firestore** ซึ่งเป็น NoSQL ฐานข้อมูล และดึงข้อมูลผ่านอินเตอร์เฟซ **`Stream` (Reactive Stream Subscription)** ครับ เมื่อมีการเปลี่ยนข้อมูลในคอลเลกชันแชทหรือสถานะของเคส ข้อมูลจะถูก Push ไปยัง UI ของผู้ใช้ทุกคนที่เกี่ยวข้องในเวลาเป็นมิลลิวินาทีโดยไม่ต้องรีเฟรชหน้าจอเลยครับ
  > สำหรับโครงสร้างตารางหลักจะถูกแบ่งออกเป็น 3 คอลเลกชันสำคัญ ได้แก่:
  > 1. `/users` สำหรับเก็บโปรไฟล์และบทบาทหน้าที่สิทธิ์
  > 2. `/incidents` สำหรับเก็บพิกัด GPS, ประเภทภัย 6 หมวด และสถานะเหตุการณ์
  > 3. Sub-collection ย่อยชื่อ `/messages` ภายใต้เหตุการณ์นั้นๆ เพื่อควบคุมประวัติแชทเฉพาะรายคดีไม่ปะปนกันครับ"

### คำถามที่ 4: การแจ้งเตือนด่วน (Push Notification) ทำงานอย่างไร และแก้ปัญหาความหน่วงได้อย่างไร?
* **แนวคำตอบ:**
  > "เราใช้บริการ **Firebase Cloud Messaging (FCM)** ร่วมกับ **Cloud Functions** ที่เขียนด้วย Node.js ครับ เมื่อผู้ประสบเหตุกด SOS เอกสารจะถูกสร้างขึ้นใน Firestore ซึ่งจะสั่งรัน Cloud Functions หลังบ้านโดยอัตโนมัติ (Database Trigger) เพื่อส่ง Multicast Notification ไปยังอุปกรณ์มือถือของกู้ภัยทุกคนทันที
  > วิธีแก้ปัญหาความหน่วงคือเราเลือก Deploy ระบบ Cloud Functions นี้ไว้ที่โซน **`asia-southeast1` (ประเทศสิงคโปร์)** ซึ่งเป็นเซิร์ฟเวอร์ที่อยู่ใกล้ประเทศไทยมากที่สุด ทำให้ส่งแจ้งเตือนได้ภายในระยะเวลาหลักมิลลิวินาที และส่งเสียงไซเรนเตือนเจ้าหน้าที่กู้ภัยได้ทันท่วงทีแม้จะปิดหน้าจอแอปพลิเคชันอยู่ครับ"

### คำถามที่ 5: แผนที่อัจฉริยะ (Adaptive Map) ที่พัฒนาขึ้นมา มีประโยชน์อย่างไรในเชิงธุรกิจและเทคนิค?
* **แนวคำตอบ:**
  > "เป็นวิดเจ็ตแผนที่ปรับตัวที่เราพัฒนาขึ้นมาเพื่อประเมินประเภทแพลตฟอร์มของผู้ใช้อัตโนมัติ (Compile-time Platform Switching) ครับ:
  > * **ฝั่งมือถือ (Android/iOS):** แอปจะรันบนไดรเวอร์ **Google Maps SDK** ทำให้เลื่อนและซูมแผนที่ได้เสถียรลื่นไหลตามเซนเซอร์มือถือของเจ้าหน้าที่และผู้แจ้งเหตุ
  > * **ฝั่งเว็บเบราว์เซอร์ (Web Dispatcher):** แอปจะสลับไปเรนเดอร์ผ่าน **`flutter_map` ร่วมกับ OpenStreetMap (OSM)** ซึ่งเป็นแผนที่เสรีไม่มีค่าใช้จ่าย API
  > ประโยชน์เชิงธุรกิจคือทำให้มหาวิทยาลัยประหยัดงบประมาณค่าใช้บริการเรียกใช้แผนที่ได้อย่างมหาศาล และไม่มีข้อจำกัดเรื่องโควตาแผนที่บนหน้าจอแสดงผลขนาดใหญ่ของศูนย์วิทยุกลางครับ"

### คำถามที่ 6: การคำนวณและประมวลผลสถิติความพึงพอใจ UAT ในระบบทำอย่างไร?
* **แนวคำตอบ:**
  > "ในส่วนแดชบอร์ดผู้ดูแลระบบ (Admin Panel) เมื่อกู้ภัยแก้ไขเคสเสร็จสิ้น (`RESOLVED`) ผู้แจ้งจะสามารถให้คะแนนความพึงพอใจการปฏิบัติการได้ ระบบหน้าจอจะทำการดึงข้อมูลคะแนนทั้งหมดแบบเรียลไทม์มาคำนวณหาค่าเฉลี่ย (**Mean**) และค่าส่วนเบี่ยงเบนมาตรฐาน (**Standard Deviation - SD**) โดยตรงจากฝั่งแอปพลิเคชัน และนำเสนอผลลัพธ์การประเมินแยกตามหมวดหมู่ภัยด้วยแผนภูมิแท่ง (Bar Chart) เพื่อวิเคราะห์ประสิทธิภาพการกู้ภัยในแต่ละประเภทได้อย่างแม่นยำครับ"
