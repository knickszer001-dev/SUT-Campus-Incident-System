# SUT Campus Incident System — คู่มือวิเคราะห์โครงสร้างซอร์สโค้ดฉบับย่อ (Workspace Lean Guide)

เอกสารฉบับนี้จัดทำขึ้นในรูปแบบคู่มือเชิงโครงสร้างเพื่อแยกแยะระหว่าง **ซอร์สโค้ดหลักของแอปพลิเคชัน (Core Production Application)** ออกจากไฟล์ที่ใช้ในขั้นตอนการพัฒนา (Developmental Artifacts), ไฟล์แบ็กอัป (Backups), โฟลเดอร์สำหรับทดลองเขียนโค้ด (Experimental Directories) และโครงสร้างการตั้งค่าของแต่ละแพลตฟอร์มในระบบพื้นที่ทำงานนี้

> [!NOTE]  
> การวิเคราะห์ในเอกสารนี้มีวัตถุประสงค์เพื่อช่วยให้อาจารย์ที่ปรึกษามหาวิทยาลัย ผู้ตรวจสอบซอร์สโค้ด และนักพัฒนารุ่นถัดไป สามารถโฟกัสไปที่ตรรกะทางธุรกิจหลัก (Core Business Logic) ของระบบจัดการเหตุฉุกเฉินภายในมหาวิทยาลัย (SUT Campus Incident System) ได้อย่างตรงจุด โดยไม่ต้องเสียเวลากับไฟล์ขยะหรือไฟล์การตั้งค่าทางเทคนิคทั่วไป

---

## 1. อะไรคือแอปพลิเคชัน "ตัวจริง" (The "REAL" App)?

แอปพลิเคชันระบบจัดการเหตุฉุกเฉินและแจ้งเหตุภายในมหาวิทยาลัยที่ทำงานได้จริง ประกอบด้วยไคลเอนต์แบบข้ามแพลตฟอร์ม (รองรับทั้ง Web, Android, iOS และ Windows) ที่พัฒนาด้วย **Flutter** เชื่อมต่อกับระบบเซิร์ฟเวอร์เลสหลังบ้าน (Serverless Backend) ที่ขับเคลื่อนด้วย **Firebase** (Cloud Firestore, Firebase Auth, Firebase Storage และ Cloud Functions)

โฟลเดอร์และไฟล์ต่อไปนี้ถือเป็น **แอปพลิเคชันหลักที่จำเป็นอย่างยิ่ง (REAL App)** ในการรันระบบและห้ามตัดออก:

| โครงสร้างไฟล์ / โมดูลหลัก | วัตถุประสงค์และความรับผิดชอบ | เหตุผลที่จำเป็นอย่างยิ่ง |
| :--- | :--- | :--- |
| **`lib/`** | ซอร์สโค้ดหลักของแอปพลิเคชัน Flutter | บรรจุตรรกะฝั่งไคลเอนต์ทั้งหมด (Front-end Logic), โมเดลข้อมูล, เซอร์วิสการประมวลผลระบบ, โครงสร้างสถาปัตยกรรม Clean Architecture แยกตามฟีเจอร์ และ UI แบบตอบสนอง (Responsive UI) |
| **`pubspec.yaml`** | ไฟล์ควบคุมแพ็กเกจและการตั้งค่าของโปรเจกต์ | กำหนดเวอร์ชันสภาพแวดล้อม (Environment SDK), รายการไลบรารีภายนอกที่จำเป็นทั้งหมด (Dependencies), การเชื่อมโยงไฟล์รูปภาพโลโก้และไฟล์เสียงเอฟเฟกต์แจ้งเตือน |
| **`functions/`** | ซอร์สโค้ด Node.js Firebase Cloud Functions | โค้ดหลังบ้านแบบเซิร์ฟเวอร์เลส (Serverless Backend) ประกอบด้วยระบบตรวจสอบตัวตน, ระบบรีเซ็ตรหัสผ่านผ่านโทรศัพท์ และระบบยิงข้อความแจ้งเตือน (FCM Push Notification) ไปยังอุปกรณ์ผู้ใช้ |
| **`firestore.rules`** | กฎความปลอดภัยของฐานข้อมูล Firestore | ไฟล์ควบคุมสิทธิ์และความปลอดภัยในการจัดการข้อมูลบนคอลเลกชันหลักทั้งหมด (`users`, `incidents`, `direct_messages`, `app_config`) เพื่อป้องกันการปลอมแปลงตัวตนและสิทธิ์การอ่านเขียนข้อมูลที่ไม่ได้รับอนุญาต |
| **`storage.rules`** | กฎความปลอดภัยของการจัดเก็บไฟล์ Cloud Storage | ทำหน้าที่ตรวจสอบความถูกต้องและประเภทไฟล์รูปภาพแนบเหตุฉุกเฉิน (`image/*`) และจำกัดขนาดรูปภาพสูงสุดไม่เกิน $10\text{ MB}$ ต่อไฟล์เพื่อป้องกันการโจมตีระบบ |
| **`assets/`** | ไฟล์สื่อการสื่อสาร โลโก้ และเสียง | บรรจุเสียงเอฟเฟกต์แจ้งเตือนเหตุฉุกเฉิน (`sounds/alert.mp3`) และไฟล์โลโก้ทางการของมหาวิทยาลัยและหน่วยงานเพื่อสร้างประสบการณ์ใช้งานระดับพรีเมียม |

## 2. ไฟล์ที่ไม่จำเป็น / สิ่งรบกวนใน Workspace

ในระหว่างขั้นตอนการพัฒนาและทดสอบระบบ จะมีไฟล์ชั่วคราวและไฟล์การตั้งค่าของตัวเขียนโค้ด (Editor Metadata) ถูกสร้างขึ้นมาโดยอัตโนมัติ ซึ่งผู้ตรวจสอบซอร์สโค้ด **สามารถมองข้ามได้อย่างปลอดภัย** ดังนี้:

### ก. ไฟล์ที่ข้ามได้ทันที (ข้อมูลสภาพแวดล้อมและการตั้งค่าของโปรแกรมเขียนโค้ด)
*   **`.metadata`**: สร้างโดย Flutter เพื่อติดตามข้อมูลช่องทางรีลีสและคุณสมบัติพื้นฐานของโปรเจกต์
*   **`mobile_user_app.iml` / `sut_campus_incident_system.iml`**: ไฟล์ตั้งค่าพื้นที่ทำงานของโปรแกรมในเครือ JetBrains / IntelliJ IDEA
*   **`.flutter-plugins-dependencies`**: ไฟล์สร้างขึ้นโดยอัตโนมัติเพื่อแสดงการทำงานของปลั๊กอินในแต่ละแพลตฟอร์ม
*   **`.gitignore`**: ไฟล์กำหนดขอบเขตเพื่อไม่ให้แคชของการคอมไพล์ ไฟล์ติดตั้ง หรือคีย์ส่วนตัวถูกอัปโหลดขึ้นสู่ระบบเก็บโค้ด (VCS Git)

### ข. ไฟล์ประวัติการพัฒนา, ตัวอย่างทดสอบ และโฟลเดอร์สำหรับ Build
*   **`cors.json`**: ไฟล์กำหนดค่าแชร์ทรัพยากรข้ามแหล่งกำเนิด (CORS) สำหรับทดสอบอัปโหลดรูปภาพบนเบราว์เซอร์
*   **`test/widget_test.dart`**: ไฟล์โค้ดทดสอบแบบจำลองวิดเจ็ตเริ่มต้นของ Flutter ซึ่งเป็นเพียงหน้าว่าง สามารถมองข้ามได้ในการประเมินสถาปัตยกรรม
*   **`android/` / `ios/` / `web/`**: โฟลเดอร์เหล่านี้คือสภาพแวดล้อมสำหรับการสร้างผลลัพธ์ของแต่ละแพลตฟอร์ม (Build Wrappers) ซึ่งจะบรรจุไฟล์ควบคุมระบบปฏิบัติการและไฟล์นำเข้าปลั๊กอินของแพลตฟอร์มนั้น ๆ แต่ตัวตรรกะทางธุรกิจหลักทั้งหมดจะถูกดึงมาจากโฟลเดอร์ `lib/` เพียงแห่งเดียวเท่านั้น

```
📁 Root Workspace (ภาพรวมจุดที่ข้ามได้)
├── 📁 .git/                         -> ข้ามได้ (ประวัติการเก็บเวอร์ชัน)
├── 📁 .dart_tool/                   -> ข้ามได้ (แคชสำหรับตัววิเคราะห์ภาษา Dart)
├── 📁 .idea/                        -> ข้ามได้ (ค่าคอนฟิกพื้นที่ทำงานของโปรแกรม IntelliJ/VS Code)
├── 📁 build/                        -> ข้ามได้ (ผลลัพธ์การแปลงโค้ดและแคชหลังการรัน)
├── 📁 test/                         -> ข้ามได้ (ไฟล์ทดสอบเริ่มต้นที่เป็นหน้าว่าง)
├── 📄 pubspec.lock                  -> สร้างอัตโนมัติ (รายการยึดตรึงเวอร์ชันของแพ็กเกจ)
└── 📄 *.iml                         -> ข้ามได้ (ไฟล์กำหนดค่าโปรเจกต์ของ IDE)
```

## 3. โครงสร้างซอร์สโค้ดฉบับย่อขั้นสูง (Lean Workspace Layout)

เพื่ออำนวยความสะดวกในการตรวจประเมินของอาจารย์ที่ปรึกษา นี่คือแผนภาพโครงสร้างซอร์สโค้ดที่แสดงเฉพาะส่วนที่เป็นตรรกะและฟังก์ชันการแจ้งเหตุของมหาวิทยาลัยโดยแท้จริง:

```
SUT-Campus-Incident-System/
├── 📁 assets/                          # ไฟล์ทรัพยากรของแอปพลิเคชัน
│   ├── 📁 images/                      # ไฟล์โลโก้ตราสัญลักษณ์ SUT และหน่วยงาน
│   └── 📁 sounds/                      # เสียงแจ้งเตือนความถี่สูง (alert.mp3)
│
├── 📁 functions/                       # โค้ดหลังบ้านระบบเซิร์ฟเวอร์เลส (Node.js)
│   ├── 📄 index.js                     # โค้ดคลาวด์ฟังก์ชันที่ติดตั้งบนเซิร์ฟเวอร์สิงคโปร์
│   └── 📄 package.json                 # ไลบรารีการเชื่อมโยงของระบบหลังบ้าน Node.js
│
├── 📁 lib/                             # ซอร์สโค้ดหลักของแอปพลิเคชันฝั่งไคลเอนต์ (Flutter)
│   ├── 📄 main.dart                    # จุดเริ่มต้นบูตแอปพลิเคชันและด่านตรวจสอบสิทธิ์ (Auth Gate)
│   │
│   ├── 📁 core/                        # รากฐานระบบ ตัวช่วยการทำงาน และการตั้งค่าส่วนกลาง
│   │   ├── 📄 adaptive_map_widget.dart # ตัวแปลงแผนที่ข้ามแพลตฟอร์ม (Leaflet Web / Google Maps Mobile)
│   │   ├── 📄 providers.dart           # ศูนย์รวม Riverpod ที่ทำหน้าที่เป็นตัวจัดการสเตตแบบไลฟ์สตรีม
│   │   ├── 📄 theme.dart               # ชุดข้อมูลสีส้มประจำมหาวิทยาลัย (SUT Orange) และฟอนต์ Outfit
│   │   └── 📄 web_notification.dart    # ด่านเลือกอินเทอร์เฟซแจ้งเตือนแยกตามระบบปฏิบัติการ
│   │
│   ├── 📁 models/                      # สัญญาข้อมูลสำหรับการแปลงออบเจกต์ (Data Models)
│   │   ├── 📄 incident_model.dart      # โครงสร้างออบเจกต์ของเหตุฉุกเฉินและลำดับเวลาปฏิบัติงาน
│   │   └── 📄 user_model.dart          # โครงสร้างโปรไฟล์ผู้ใช้งานและพิกัดผู้ระงับเหตุ
│   │
│   └── 📁 features/                    # ส่วนการจัดการระบบแยกตามฟีเจอร์การใช้งาน
│       ├── 📁 auth/                    # ระบบลงทะเบียนและระบบแปลงรหัสผ่านเป็นอีเมลเสมือน
│       ├── 📁 chat/                    # ห้องสนทนากลุ่มตามเหตุการณ์และการส่งข้อความส่วนตัว (DM)
│       ├── 📁 dispatcher/              # แผงควบคุมหลัก แผนที่สั่งการ และการจัดการภาระงานผู้ระงับเหตุ
│       ├── 📁 home/                    # หน้าหลักของนักศึกษา การเลือกประเภทภัย และฟอร์มแจ้งเหตุด่วน
│       ├── 📁 incident/                # กฎการเปลี่ยนสเตตเหตุการณ์, ระบบคำนวณคะแนน F5 และระบบติดตาม F6
│       ├── 📁 notification/            # การประมวลผลเบื้องหลังของระบบส่งข้อความแจ้งเตือน (FCM)

└── 📄 storage.rules                    # ตัวกรองการอัปโหลดไฟล์รูปภาพ Storage
```

## 4. แผนภาพลำดับการทำงานและสถาปัตยกรรม (Execution Flow)

ภาพรวมของวิธีการรันระบบไคลเอนต์และการปฏิสัมพันธ์กับฐานข้อมูลหลังบ้าน

### ก. ลำดับการเริ่มต้นแอปและการนำทาง (App Startup & Routing Pipeline)
```mermaid
graph TD
    A[main.dart: main] -->|1. โหลด Firebase และระบบแจ้งเตือนท้องถิ่น| B(main.dart: ProviderScope)
    B -->|2. ตรวจสอบสตรีมสถานะผู้ใช้งาน| C{มีประวัติล็อกอินค้างไว้หรือไม่?}
    C -->|ไม่มีบัญชีค้างอยู่| D[auth/presentation/login_screen.dart]
    C -->|มีบัญชีค้างอยู่| E[auth/presentation/role_redirect.dart]
    E -->|3. ตรวจสอบและสร้างฐานข้อมูลเริ่มต้นด้วย ensureUserProfile| F[ดึงข้อมูลสิทธิ์จากคอลเลกชัน /users/uid]
    F -->|dispatcher (ศูนย์รับเหตุ)| G[dispatcher/dispatcher_screen.dart]
    F -->|responder (ผู้รับเคส)| H[responder/responder_dashboard.dart]
    F -->|admin (ผู้ดูแลระบบ)| I[admin/admin_dashboard.dart]
    F -->|user (นักศึกษา/บุคลากร)| J[home/home_screen.dart]
```

### ข. ระบบจัดการสถานะข้อมูล (State Management System)
*   **ใช้ไลบรารี Riverpod**: เพื่อแยกตรรกะทางธุรกิจออกจาก UI โดยเชื่อมต่อกับกระแสข้อมูลแบบเรียลไทม์ (Live Streams) ของ Firebase
*   **การอัปเดตสิทธิ์อัตโนมัติ**: `authStateProvider` ทำหน้าที่คอยตรวจจับเซสชันการเข้าสู่ระบบ ขณะที่ `currentUserProvider` คอยตรวจจับการเปลี่ยนแปลงของเอกสารโปรไฟล์ใน Firestore `/users/{uid}` ส่งผลให้เมื่อข้อมูลผู้ใช้ (เช่น คะแนนจิตอาสา หรือสิทธิ์เข้าถึง) เปลี่ยนแปลง ตัวหน้าจอ UI ในทุกตำแหน่งของแอปพลิเคชันจะปรับปรุงการแสดงผลให้สอดคล้องกันทันทีโดยไม่ต้องทำการรีเฟรชหน้าจอ

### ค. ลำดับการแจ้งเหตุและการส่งข้อความแจ้งเตือน (Incident Submission & Notifications)
1.  **การยื่นคำร้อง (Submission)**: นักศึกษาแจ้งเหตุผ่านหน้าจอโปรเจกต์ `lib/features/home/home_screen.dart` ซึ่งจะไปเรียกเมธอด `submitIncident()` ในคลาส `IncidentRepository`
2.  **การสั่งยิงข้อความ (Alert Dispatch)**: ตัวควบคุม `IncidentRepository` จะส่งคำร้องไปยังระบบคลาวด์ฟังก์ชัน `sendPushNotification` เพื่อติดต่อกับ FCM Gateway
3.  **การเลือกช่องทางรับข้อความ (FCM Routing)**: อุปกรณ์จะรับสัญญาณตามคีย์ช่องแจ้งเตือน (`urgent_incidents_v2` สำหรับเหตุใหม่, `chat_messages` สำหรับสตรีมแชท)
4.  **ระบบประมวลผลเรียลไทม์ (Real-time Listening)**:
    *   **บนมือถือระบบ Native**: ตัวจัดการ FCM ไอโซเลตจะวิเคราะห์เพย์โหลดและแสดงแถบหน้าต่างป๊อปอัปแจ้งเตือนผ่านปลั๊กอิน `flutter_local_notifications` พร้อมเล่นเสียงไซเรนจากเครื่อง
    *   **บนเบราว์เซอร์ Web**: คลาส `WebNotificationWatcher` จะจับสัญญาณความเปลี่ยนแปลงเอกสาร Firestore โดยตรง เพื่อสั่งเปิดเบราว์เซอร์แจ้งเตือน (Browser Native Notification) เล่นเสียงสัญญาณสั้นผ่านระบบ Web Audio API พร้อมแสดงแผงข้อความเลื่อนขึ้นด้านล่าง (Overlay Entry) ที่จะซ่อนตัวเองโดยอัตโนมัติภายใน 6 วินาที
