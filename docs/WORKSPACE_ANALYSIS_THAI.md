# คู่มือวิเคราะห์โครงสร้างระบบและการเริ่มต้นพัฒนา (Workspace Analysis & Developer Onboarding Guide)

เอกสารฉบับนี้จัดทำขึ้นในรูปแบบภาษาไทยเชิงเทคนิค โดยมีวัตถุประสงค์เพื่อวิเคราะห์โครงสร้างซอร์สโค้ด แพลตฟอร์ม และตรรกะระบบควบคุมความปลอดภัย **SUT Campus Incident System** อย่างละเอียด เพื่อช่วยให้นักพัฒนาหน้าใหม่ คณะกรรมการตรวจสอบ หรือผู้ควบคุมโปรเจกต์สามารถทำความเข้าใจแผนภาพรวมและเข้าถึงส่วนต่าง ๆ ของคลังซอร์สโค้ดได้อย่างถูกต้องและเป็นระบบ

## 1. บทสรุปผู้บริหาร (Executive Summary)

### วัตถุประสงค์หลักของระบบ (System Purpose)
**SUT Campus Incident System** คือแอปพลิเคชันจัดการและประสานงานระงับเหตุภัยฉุกเฉินแบบเรียลไทม์ภายในวิทยาเขต **มหาวิทยาลัยเทคโนโลยีสุรนารี (มทส.)** โดยระบบได้รับการพัฒนาขึ้นเพื่อทำหน้าที่เชื่อมต่อและประสานงานข้อมูลอย่างเป็นลำดับขั้นระหว่างกลุ่มบทบาทผู้ใช้งาน 3 ฝ่ายหลัก ได้แก่:
1.  **นักศึกษาและบุคลากร (Reporters)**: แจ้งเหตุด่วนพร้อมพิกัดดาวเทียม รายละเอียดข้อความ และรูปภาพสด ณ จุดเกิดภัย
2.  **เจ้าหน้าที่ควบคุมศูนย์ประสานงานรักษาความปลอดภัย (Dispatchers)**: ตรวจทานคดีความถี่สูงผ่านแผงวิทยุสั่งการ คัดกรองและประเมินภาระงานเพื่อส่งทีมเข้าช่วยเหลือ
3.  **เจ้าหน้าที่หน่วยระงับเหตุฉุกเฉิน (Responders)**: รับมอบงาน เดินทาง นำทางพิกัดดาวเทียม สนทนาและอัปเดตสเตตัสการช่วยเหลือ

### เทคโนโลยีหลักที่ใช้ในโครงการ (Technologies Used)
*   **โครงสร้างหน้าบ้าน (Client UI)**: พัฒนาด้วยเฟรมเวิร์ก Flutter (ขอบเขตคอมไพเลอร์ Dart SDK `^3.10.1` รองรับขยายสภาพแวดล้อมตั้งแต่ `>=3.10.0` ขึ้นไป)
*   **ชุดสเตตควบคุมระบบ (State Management)**: เรียกใช้ไลบรารี Riverpod (`v2.5.1`) ควบคุมแบบ Stream ในชั้นแคชข้อมูลส่วนกลางเพื่อประสานงานการไหลเวียนข้อมูลสด
*   **บริการเซิร์ฟเวอร์เลสหลังบ้าน (Backend Serverless)**: ทำงานบนแพลตฟอร์ม Firebase เต็มระบบ ได้แก่:
    *   **Firebase Auth**: จัดการเซสชันล็อกอินและจำลองโดเมนบัญชีวิทยาเขต
    *   **Cloud Firestore**: เชื่อมส่งข้อมูลแบบ NoSQL จัดเก็บข้อมูลประวัติคดี แชทกลุ่ม แชทส่วนตัว และตำแหน่งภูมิศาสตร์เรียลไทม์
    *   **Firebase Storage**: พื้นที่คลาวด์สำหรับรับฝากรูปภาพความละเอียดสูงประกอบรายงานแจ้งเหตุ
    *   **Cloud Functions**: รันโค้ดประมวลผลฝั่งเซิร์ฟเวอร์เลสหลังบ้าน (Node.js) ติดตั้งในภูมิภาค **สิงคโปร์ (`asia-southeast1`)** เพื่อยิงสัญญาณแจ้งเตือนระบบ (FCM) และดำเนินตรรกะรีเซ็ตรหัสผ่านแบบพิเศษ
    *   **Firebase Crashlytics & Remote Config**: ดักจับบั๊กความล้มเหลวอัตโนมัติบนอุปกรณ์และจัดการตรวจเช็คเทียบเลขรุ่นโปรแกรม
*   **เทคโนโลยีแผนที่พิกัดดาวเทียม (Geospatial & Maps)**:
    *   **ระบบ Native สมาร์ตโฟน (Android/iOS)**: ควบคุมโดย Google Maps SDK (`google_maps_flutter`) เพื่อการแสดงผลแผนที่เวกเตอร์ความลื่นไหลสูง
    *   **ระบบ Web และ Windows**: ใช้ Leaflet Tile Layer ในแอปผ่านชุดคำสั่ง `flutter_map` ร่วมกับแผนที่ฟรี OpenStreetMap (OSM) เพื่อตัดภาระต้นทุนค่าเรียกใช้ API แผนที่บนบราวเซอร์
    *   **การจับสัญญาณพิกัด**: เรียกดึงพิกัดผ่าน Geolocator API (`geolocator`) โดยกำหนดฟิลเตอร์ตรวจจับความเคลื่อนไหวทุกระยะ 10 เมตร

## 2. ภาพรวมไดเรกทอรีและโฟลเดอร์ในระบบ (Workspace Structure Overview)

โครงสร้างโฟลเดอร์หลักภายในพื้นที่โครงการ SUT Campus Incident System:

### โฟลเดอร์: `/lib`
*   **ประเภท**: ซอร์สโค้ดแอปพลิเคชันหลัก (Application Source Code)
*   **วัตถุประสงค์**: บรรจุชุดคำสั่งภาษา Dart ทั้งหมด ตั้งแต่วิดเจ็ต UI หน้าจอ, การกำหนดสไตล์ความสวยงาม, ไฟล์ควบคุม State, ไปจนถึงคลาสประสานงานข้อมูลฝั่งเซิร์ฟเวอร์
*   **ความเชื่อมโยง**: นำเข้าตราสัญลักษณ์และไฟล์เสียงแจ้งเตือนจาก `/assets` และเรียกเปิดค่าคอนฟิกสิทธิ์การเชื่อมต่อผ่านค่าอ้างอิง Firebase

### โฟลเดอร์: `/assets`
*   **ประเภท**: โฟลเดอร์จัดเก็บทรัพยากรระบบ (Asset Directory)
*   **วัตถุประสงค์**: เก็บรวบรวมไฟล์ภายนอกที่แอปต้องการเรียกใช้ในขั้นตอนแสดงผล เช่น:
    *   `assets/images/`: ภาพโลโก้มหาวิทยาลัย (`university_logo.png`, `sut_incident_logo.png`)
    *   `assets/sounds/`: ไฟล์เสียงแจ้งเตือนไซเรนด่วน (`alert.mp3`)
*   **ความเชื่อมโยง**: ลงทะเบียนพาธผ่านไฟล์ `pubspec.yaml` เพื่อเปิดให้ชุดคำสั่งใน `/lib` นำรูปไปวาดและเล่นเสียงแจ้งเตือน

### โฟลเดอร์: `/docs`
*   **ประเภท**: เอกสารอ้างอิงและคำอธิบายระบบ (Documentation)
*   **วัตถุประสงค์**: พื้นที่จัดเก็บเอกสารเชิงเทคนิคเพื่อรองรับการประเมินและการซ่อมบำรุงซอฟต์แวร์ในระยะยาว:
    *   `architecture.md`: เจาะลึกเลเยอร์สถาปัตยกรรมและกระบวนการทำงานของระบบ suppression ดักเสียงแจ้งเตือนซ้ำซ้อน
    *   `database.md`: สเปกตารางเอกสาร ข้อมูลฟิลด์ย่อยใน NoSQL และกฎความปลอดภัย
    *   `features.md`: รายละเอียดฟังก์ชันระบบ F1 ถึง F10
    *   `lean_audit_report.md`: รายงานผลวิเคราะห์ความคุ้มค่าและความย่อย่นของตัวโค้ดโครงการ
*   **ความเชื่อมโยง**: เป็นเอกสารข้อมูลอ้างอิงของวิศวกรซอฟต์แวร์ในการทำความเข้าใจการเชื่อมต่อของไฟล์ใน `/lib` และตัวฟังก์ชันใน `/functions`

### โฟลเดอร์: `/functions`
*   **ประเภท**: ซอร์สโค้ดและข้อมูลหลังบ้านเซิร์ฟเวอร์เลส (Backend Source Code - Node.js)
*   **วัตถุประสงค์**: รวบรวมสคริปต์ JavaScript / Node.js สำหรับนำขึ้นรันเป็น Cloud Functions
*   **ความเชื่อมโยง**: ถูกเข้าถึงความปลอดภัยฝั่งหลังบ้านโดยคำสั่ง HTTPS Callable จากฝั่งแอปไคลเอนต์

### โฟลเดอร์: `/test`
*   **ประเภท**: โฟลเดอร์ชุดกรณีทดสอบระบบ (Test Suite)
*   **วัตถุประสงค์**: บรรจุไฟล์สำหรับดำเนินการจำลองการทำงานโปรแกรม (`widget_test.dart`)
*   **ความเชื่อมโยง**: รันสอบทานเทียบความถูกต้องทางโครงสร้างของแอปหลักใน `/lib` เพื่อป้องสถานการณ์ระบบล่มหลังคอมไพล์

### โฟลเดอร์ฝั่ง Native แพลตฟอร์ม (`/android`, `/ios`, `/web`, `/windows`)
*   **ประเภท**: ค่านำส่งระดับระบบปฏิบัติการ (Tooling / Platform Configuration)
*   **วัตถุประสงค์**: เก็บเอกสารนำส่งสิทธิ์เข้าถึงอุปกรณ์ของโทรศัพท์ (เช่น การขอใช้พิกัดจีพีเอส สิทธิ์เปิดไมโครโฟนหรือกล้องถ่ายรูป) และจัดตั้งหน้าต่างเว็บ
*   **ความเชื่อมโยง**: แปลงรหัสซอร์สโค้ด Dart ในโฟลเดอร์ `/lib` ให้ออกมาเป็นแพลตฟอร์มเป้าหมายของโทรศัพท์มือถือและหน้าเบราว์เซอร์อย่างสมบูรณ์

## 3. การวิเคราะห์รายละเอียดเทคโนโลยีและโมดูลเชื่อมโยง (Technology & Dependency Analysis)

ตัวไฟล์คอนฟิก `pubspec.yaml` มีการติดตั้งระบบไลบรารีเพื่อรองรับการทำงานต่าง ๆ ดังนี้:

*   **`flutter_riverpod` (`^2.5.1`)**: ทำหน้าที่จัดการ State Management ในระดับสากล ช่วยแยกส่วนควบคุมตรรกะระบบ (Business Logic) ออกจากความสวยงามของวิดเจ็ตหน้าจอ UI ได้อย่างสมบูรณ์แบบ
*   **`cloud_firestore` (`^6.1.3`)**: ดูแลช่องทางการไหลข้อมูลแบบเรียลไทม์ โดยคอยดึงเอกสารบันทึกเหตุการณ์ฉุกเฉินและแชทติดต่อกันในระบบคดีมาซิงค์ลงสู่อินเทอร์เฟซผู้ใช้อย่างถูกต้อง
*   **`firebase_auth` (`^6.2.0`)**: คัดกรองและควบคุมกระบวนการล็อกอิน ตรวจทานเซสชันบัญชีผู้ใช้
*   **`firebase_storage` (`^13.1.0`)**: ดำเนินขั้นตอนการโอนส่งรูปภาพสดจุดรับแจ้งคดีขึ้นเซิร์ฟเวอร์คลาวด์ พร้อมส่งคืนลิงก์ปลอดภัยกลับมาเพื่อนำไปแนบเป็นรูปเกิดเหตุ
*   **`firebase_messaging` (`^16.1.2`)**: ดูแลการลงทะเบียน FCM Token ประจำตัวเครื่อง และรับข้อความแจ้งภัยป๊อปอัปบนโทรศัพท์
*   **`google_maps_flutter` (`^2.6.1`)**: โหลดและเรนเดอร์กราฟิกแผนที่ความคมชัดสูงสำหรับอุปกรณ์ Native
*   **`flutter_map` (`^7.0.2`)** ร่วมกับ **`latlong2` (`^0.9.1`)**: วาดแผนที่ Leaflet ในหน้าจอบราวเซอร์ Web ป้องกันการชาร์จต้นทุนค่าแผนที่ API ของกูเกิล
*   **`geolocator` (`^11.0.0`)**: ควบคุมสิทธิ์การเรียกอ่านพิกัดเชิงภูมิศาสตร์จากดาวเทียมผ่านอุปกรณ์ผู้ใช้
*   **`audioplayers` (`^6.1.0`)**: เล่นไฟล์เสียงเตือนภัยไซเรนด่วนและสัญญาณเตือนเสียงความถี่สูงบนอุปกรณ์ Native
*   **`fl_chart` (`^0.68.0`)**: วาดแผนภาพกราฟแท่งและกราฟสถิติคดีเกิดภัยสะสมในมหาวิทยาลัย มทส. ให้ศูนย์สั่งการได้วิเคราะห์ข้อมูล
*   **`introduction_screen` (`^3.1.14`)**: จัดหน้าจอสไลด์นำเสนอข้อแนะนำและวิธีกรอกข้อมูลแจ้งเหตุฉุกเฉินสำหรับผู้ใช้แอปครั้งแรก

## 4. วิเคราะห์การออกแบบสถาปัตยกรรม (Architecture & Design Pattern Analysis)

โปรเจกต์นี้ได้รับการพัฒนาขึ้นโดยยึดตามหลักสถาปัตยกรรมทางวิศวกรรมซอฟต์แวร์ระดับสูง:

### การวิเคราะห์สถาปัตยกรรมแบบ Clean Architecture
*   **เลเยอร์นำเสนอข้อมูล (Presentation Layer - UI & Controllers)**: ทำหน้าที่เรนเดอร์อินเทอร์เฟซด้วยส่วนประกอบที่สวยงาม เข้ามาตรวจดูความเปลี่ยนแปลงจาก Riverpod Providers (เช่น `currentUserProvider`, `authStateProvider`) และประเมินเพื่ออัปเดตข้อมูลขึ้นหน้าจอ
*   **เลเยอร์ตรรกะทางธุรกิจ (Domain Layer - Business Logic & Rules)**: อยู่ภายในคลาส `/lib/features/incident/domain/responder_logic.dart` คอยจัดการและตรวจสอบกฎในการปรับสเตตัสต่าง ๆ เช่น ฟังก์ชัน `ResponderLogic.canTransition` เพื่อดักจับและอนุญาตให้เปลี่ยนสถานะของเหตุการณ์เฉพาะรายการที่ถูกต้องเท่านั้น
*   **เลเยอร์จัดการข้อมูล (Data Layer - Repositories & Adapters)**: ปรากฏในกลุ่มคลาสของ Repository เช่น `IncidentRepository` และ `AuthRepository` ทำหน้าที่แปลงเอกสาร JSON เป็นออบเจกต์ภาษา Dart และเรียกเปิดเครื่องมือส่งผ่านเครือข่าย

### รูปแบบแผนการออกแบบที่ใช้ในระบบ (Design Patterns)
1.  **Repository Pattern**: แยกคำสั่งสำหรับดึงข้อมูลและ API ออกจากปุ่มกด UI ทำให้ในอนาคตหากต้องการย้ายระบบเซิร์ฟเวอร์หรือฐานข้อมูล ผู้พัฒนาจะสามารถปรับจูนได้ง่ายโดยไม่ต้องไปยุ่งเกี่ยวกับรหัสในหน้าจอแสดงผล
2.  **Adapter Pattern (`AdaptiveMapWidget`)**: นำมาประยุกต์ใช้เพื่อผสานอินเทอร์เฟซแผนที่ต่างระบบ โดยกำหนดให้ตัวแอปสร้างพอร์ตหน้ากากควบคุมแผนที่ชุดเดียวกัน แต่ตัวระบบจะแปลงโครงสร้างไปเรียก Google Maps หรือ Leaflet ให้อัตโนมัติในตอนคอมไพล์
3.  **State Notification Streams**: รวมชุด StreamProviders ในไฟล์ระบบ `providers.dart` เพื่อสร้างท่อซิงค์ข้อมูลเรียลไทม์จากฐานข้อมูล Firestore ยิงตรงสู่หน้าจอ

## 5. การจำแนกโมดูลการทำงานหลัก (Feature / Module Breakdown)

### 1. ระบบยืนยันตัวตนและลงทะเบียนกู้คืนบัญชี (`/lib/features/auth`)
*   **วัตถุประสงค์**: ดูแลขั้นตอนลงชื่อเข้าใช้ระบบ, ลงทะเบียนสิทธิ์ของนักศึกษา และดำเนินการรีเซ็ตรหัสผ่านผ่านเบอร์โทรศัพท์
*   **ไฟล์และไดเรกทอรีเด่น**: `login_screen.dart`, `register_screen.dart`, `role_redirect.dart`, `auth_repository.dart`
*   **กระบวนการประมวลผล**: แปลงรหัสประจำตัว มทส. (เช่น `B1234567`) ต่อท้ายหลังบ้านด้วยโดเมนมหาวิทยาลัยจำลอง (`@campus.local`) เพื่อส่งเข้าตรวจสอบสิทธิ์ใน Firebase Auth และตรวจสอบความปลอดภัยเบอร์โทรด้วยเซิร์ฟเวอร์หลังบ้าน Node.js
*   **การเชื่อมโยงระบบย่อย**: เรียกใช้งานไลบรารี `firebase_auth`, `cloud_firestore`, และ `cloud_functions`

### 2. ระบบกรอกฟอร์มแจ้งเหตุฉุกเฉิน มทส. (`/lib/features/incident`)
*   **วัตถุประสงค์**: สนับสนุนการแจ้งเหตุด่วนความเร็วสูงสำหรับนักศึกษาและเจ้าหน้าที่ภาคสนาม
*   **ไฟล์และไดเรกทอรีเด่น**: `report_incident_screen.dart`, `choose_report_type_screen.dart`, `incident_repository.dart`
*   **กระบวนการประมวลผล**: ปล่อยสัญญาณขอตรวจพิกัดปัจจุบันของผู้ใช้, นำส่งรูปภาพประกอบภัยเข้าสู่ระบบคลาวด์ Firebase Storage และสร้างประวัติตู้คดีความมั่นคงบันทึกลง Firestore ในฐานะเอกสารที่มีพิกัดแผนที่กำกับ
*   **การเชื่อมโยงระบบย่อย**: เรียกเชื่อมต่อ `geolocator`, `image_picker`, และ `firebase_storage`

### 3. ระบบวิทยุและแผนผังควบคุมสำหรับผู้ประสานงานภัย (`/lib/features/dispatcher`)
*   **วัตถุประสงค์**: เป็นศูนย์สั่งการหลัก คอยประเมินพิกัด ตรวจสอบข้อความแจ้งเตือนความถี่สูง และมอบภารกิจระงับภัย
*   **ไฟล์และไดเรกทอรีเด่น**: `dispatcher_screen.dart`, `incident_panel.dart`, `responders_panel.dart`, `sound_alert_service.dart`
*   **กระบวนการประมวลผล**: วาดหมุดคดีแจ้งภัยทั้งหมดบนแผนที่รวมขนาดใหญ่ ตรวจค้นตำแหน่งปัจจุบันของเจ้าหน้าที่ระงับเหตุฉุกเฉิน มทส. และใช้สูตรโมเดลประเมินคะแนนช่วยเหลือที่ตรงที่สุดให้เจ้าหน้าที่ศูนย์แตะเลือกสั่งมอบงาน
*   **การเชื่อมโยงระบบย่อย**: ทำงานควบคู่กับ `google_maps_flutter`, `flutter_map`, และ `fl_chart`

### 4. แผงงานสำหรับเจ้าหน้าที่เผชิญเหตุภาคสนาม (`/lib/features/responder`)
*   **วัตถุประสงค์**: ระบบควบคุมและประสานงานระงับภัยฉุกเฉินสำหรับเจ้าหน้าที่หน่วยระงับเหตุ
*   **ไฟล์และไดเรกทอรีเด่น**: `responder_dashboard.dart`, `navigator_screen.dart`
*   **กระบวนการประมวลผล**: แจ้งเตือนภารกิจใหม่ในสเตตัสล่าสุด นำทางเจ้าหน้าที่ไปยังหมุด GPS ของผู้ประสบภัย และบันทึกขั้นตอนช่วยเหลือการดำเนินงานแบบเรียลไทม์เพื่อปิดคดี
*   **การเชื่อมโยงระบบย่อย**: ซิงค์พิกัดแผนที่ผ่าน `google_maps_flutter` และคอยอัปเดตสิทธิ์ดาวเทียมด้วย `geolocator`

### 5. ท่อสนทนาเรียลไทม์เพื่อความปลอดภัย (`/lib/features/chat`)
*   **วัตถุประสงค์**: ช่องทางติดต่อประสานงานสดผ่านข้อความและรูปภาพในที่เกิดเหตุ
*   **ไฟล์และไดเรกทอรีเด่น**: `chat_screen.dart` (กลุ่มประจำเคส), `direct_chat_screen.dart` (แชทตรงเจ้าหน้าที่และศูนย์รับเรื่อง)
*   **กระบวนการประมวลผล**: ประมวลผลเอกสารข้อความผ่านระบบ nested subcollections ของ Firestore โดยแสดงลำดับแชทตามเวลาจริง
*   **การเชื่อมโยงระบบย่อย**: ขับเคลื่อนบน `cloud_firestore` และโหลดภาพเร็วด้วย `cached_network_image`

## 7. คำอธิบายเจาะลึกรายละเอียดไฟล์เด่น (File-by-File Explanation)

### ไฟล์: `lib/features/auth/data/auth_repository.dart`
*   **วัตถุประสงค์**: จัดการกระบวนการตรวจสอบสิทธิ์ ล็อกอินผ่านรหัสนักศึกษา ตรวจประวัติและสิทธิ์เข้าใช้ระบบ
*   **คลาสสำคัญ**: `AuthRepository`
*   **ฟังก์ชันการทำงานเด่น**:
    *   `signInWithStudentId(String studentId, String password)`: ทำการดักต่อท้ายรหัสนักศึกษาด้วยคำว่า `@campus.local` เพื่อนำส่งเข้าล็อกอิน Firebase Auth
    *   `registerUser({required String studentId, required String phone, ...})`: บันทึกข้อมูลทะเบียนผู้ใช้ใหม่ลงคอลเลกชัน `/users` โดยล็อกบทบาทเริ่มต้นที่ระดับ `user` (นักศึกษา/บุคลากรทั่วไป)
    *   `getCurrentUserData(String uid)`: ดึงข้อมูลโปรไฟล์ผู้ใช้งาน ณ คีย์ยูสเซอร์ปัจจุบัน
*   **ไลบรารีที่พึ่งพิง**: `firebase_auth`, `cloud_firestore`
*   **ไฟล์ที่เรียกใช้**: `login_screen.dart`, `register_screen.dart`, `providers.dart`

### ไฟล์: `lib/features/incident/data/incident_repository.dart`
*   **วัตถุประสงค์**: รับผิดชอบชุดคำสั่งเขียนและแก้ไขคดีแจ้งภัย อัปโหลดข้อมูลและภาพ รวมไปถึงสร้างสถิติคดีสะสม
*   **คลาสสำคัญ**: `IncidentRepository`
*   **ฟังก์ชันการทำงานเด่น**:
    *   `createIncident(Incident incident, List<XFile> images)`: ดำเนินขั้นตอนอัปโหลดภาพสดขึ้นเซิร์ฟเวอร์คลาวด์ พร้อมเขียนเอกสารคดีใหม่ในฐานข้อมูล
    *   `assignResponder(String incidentId, String responderId, String responderName)`: ใช้ธุรกรรมทรานเซกชัน (Firestore Transaction) เพื่อรับรองความปลอดภัยในการปรับสถานะคดีและตั้งชื่อผู้ช่วยชีวิต
    *   `getRespondersWithStats()`: รวบรวมสถิติคดีเกิดใหม่ที่ยังระงับเหตุอยู่แบบรวมคำสั่ง เพื่อสกัดปัญหาการวนคิวรีซ้ำซ้อนในฐานข้อมูล ($N+1$ query)
*   **ไลบรารีที่พึ่งพิง**: `cloud_firestore`, `firebase_storage`
*   **ไฟล์ที่เรียกใช้**: `providers.dart`, `incident_panel.dart`, `report_incident_screen.dart`

    *   `_watchDirectMessages(...)` และ `_watchStatusUpdates(...)`: รับข้อมูลเรียลไทม์ของคดีเกิดใหม่เพื่อกระตุ้นไซเรน

## 8. คำอธิบายคลาสและฟังก์ชันเด่นในระบบ (Function / Class Analysis)

### คลาส: `AuthRepository`
*   **ขอบข่ายความรับผิดชอบ**: แยกตัวแปรและ API ล็อกอิน ยืนยันสัญญารุ่นผู้ใช้ และเขียนบันทึกเอกสารสเตตัสลงคอลเลกชัน `/users`
*   **อินพุตตัวรับ**: ออบเจกต์คำสั่งเชื่อมตรง `FirebaseAuth` และ `FirebaseFirestore`
*   **พฤติกรรมการเชื่อมโยง**: แปลงค่ารหัสนักศึกษา มทส. ให้กลายเป็นโดเมนลงชื่อระบบความปลอดภัยก่อนยื่นขอใช้บริการ Firebase

### คลาส: `IncidentRepository`
*   **ขอบข่ายความรับผิดชอบ**: คลาสหลักดูแลตู้สเตตัสเหตุร้าย การปรับค่าพิกัดความเคลื่อนไหว และดำเนินธุรกรรมความปลอดภัยระดับฐานข้อมูล
*   **อินพุตตัวรับ**: ออบเจกต์เชื่อมต่อคีย์ Firestore
*   **พฤติกรรมการเชื่อมโยง**: สลับระดับความฉุกเฉิน ค้นพิกัด และยื่นเรื่องปิดประวัติคดีลงสู่คอลเลกชัน `/incidents`

### ฟังก์ชัน: `canTransition(String currentStatus, String nextStatus)`
*   **ขอบข่ายความรับผิดชอบ**: จัดการตรวจสอบและคัดกรองการเลื่อนลำดับขั้นเหตุร้าย:
    $$\text{NEW (เกิดใหม่)} \longrightarrow \text{IN\_PROGRESS (กำลังระงับภัย)} \longrightarrow \text{RESOLVED (เหตุคลี่คลาย)}$$
*   **พารามิเตอร์**: รับตัวแปร `currentStatus` (สถานะปัจจุบัน) และ `nextStatus` (สถานะถัดไป)
*   **ผลลัพธ์ที่ส่งคืน**: คืนค่า `true` หากเป็นไปตามขั้นตอนระงับภัย และคืนค่า `false` หากพยายามย้อนสถานะคดี
*   **การเชื่อมโยง**: ประกาศใช้งานเดี่ยวในชั้นสถาปัตยกรรมระดับ Domain (`ResponderLogic`) ซึ่งแยกตรรกะนี้ออกจาก UI และฐานข้อมูลอย่างอิสระ ทำให้ง่ายต่อการเขียนการทดสอบระบบ (Unit Test)

## 9. เส้นทางการไหลข้อมูลและลำดับขั้นประมวลผล (Data Flow & System Workflow)

### แผนผังเส้นทางข้อมูลการแจ้งเหตุฉุกเฉิน (Incident Reporting Flow)
```
[หน้าจอกรอกข้อมูลแจ้งเหตุ]
        │
        ▼ (อ่านข้อมูลเซ็นเซอร์ดาวเทียมจีพีเอสผ่าน geolocator)
[ตรวจรับค่าละติจูดและลองจิจูด]
        │
        ▼ (ดำเนินการส่งภาพสดแนบขึ้นระบบจัดเก็บคลาวด์)
[อัปโหลดไฟล์ขึ้น Firebase Storage]
        │
        ▼ (เขียนเอกสารคดีใหม่บันทึกลงในคอลเลกชันหลัก)
[สร้างประวัติข้อมูลคดี: /incidents/{incidentId}] (ค่าเริ่มต้น: status = 'NEW', timelineStatus = 'REPORTED')
```

### แผนผังประเมินสถิติและจับคู่งานศูนย์สั่งการ (Dispatcher Assignment Flow)
```
[ฐานข้อมูล Firestore: ดักสัญญาณคดีแจ้งใหม่]
        │
        ▼ (ยื่นคำสั่งขอประเมินภาระงานของทีมช่วยภัยฉุกเฉิน)
[รวมค่าคดีระงับเหตุสะสมลงในแคช] (ป้องกันสถิตบวนคิวรีวนซ้ำระดับฐานข้อมูล)
        │
        ▼ (คำนวณคะแนนเจ้าหน้าที่ทีมฉุกเฉิน มทส. ที่เหมาะสม)
[โมเดลคำนวณสูตรดัชนีคะแนน F5] (วิเคราะห์ค่าความเหมาะสมจาก แผนก, ภาระงานสะสม และ ระยะทางพิกัด)
        │
        ▼ (ดำเนินคำสั่งมอบงานระดับทรานเซกชันล็อค)
[ปุ่มประสานมอบงานศูนย์สั่งการ] (ปรับสถานะ status = 'IN_PROGRESS', responderId = assignedUid)
```

### แผนผังชีวิตการระงับภัยภาคสนามของเจ้าหน้าที่ (Incident Resolution Flow)
```
[แจ้งเตือนภารกิจของฉันผ่าน assignedToMeIncidentsStreamProvider]
        │
        ▼ (เจ้าหน้าที่กู้ภัยแตะรับคำสั่งระงับภัย)
[อัปเดตขั้นตอนการระงับเหตุ: 'ACCEPTED']
        │
        ▼ (เจ้าหน้าที่เริ่มออกเดินทางไปยังที่เกิดภัยฉุกเฉิน)
[อัปเดตขั้นตอนการระงับเหตุ: 'EN_ROUTE']
        │
        ▼ (เจ้าหน้าที่หน่วยกู้ชีพเดินเท้ามาถึงที่เกิดเหตุและพบผู้ป่วย)
[อัปเดตขั้นตอนการระงับเหตุ: 'ARRIVED']
        │
        ▼ (ดำเนินคดีเสร็จสิ้น แตะปิดคดีพร้อมบันทึกสรุปข้อมูลความปลอดภัย)
[ปิดเคสสมบูรณ์: status = 'RESOLVED']
```

## 10. การจำแนกสิทธิและขอบเขตชิ้นส่วนในโปรเจกต์ (Generated vs Core Source Classification)

### โค้ดหลักระบบไคลเอนต์ (Core Application Source Code - ลำดับความสำคัญในการพัฒนา: สูงที่สุด)
*   `/lib/main.dart`: จัดตู้เซอร์วิสเริ่มต้น, ติดตั้ง Firebase และเป็นประตูตรวจยืนยันสิทธิ์บทบาทผู้ใช้งานแอป
*   `/lib/core/providers.dart`: แหล่งประกาศตัวจัดการ State Management ของ Riverpod ที่ดูแลการสตรีมพิกัดและแชทแบบเรียลไทม์
*   `/lib/core/adaptive_map_widget.dart`: วิดเจ็ตสากลควบคุมชิ้นส่วนแผนที่ดาวเทียมบนทุกประเภทระบบอุปกรณ์
*   `/lib/models/`: โมเดลออบเจกต์อ้างอิงข้อมูลของคดี บุคคล และแชทประสานงาน

### ซอร์สโค้ดฟีเจอร์ระดับธุรกิจ (Feature Code - ลำดับความสำคัญในการพัฒนา: สูง)
*   `/lib/features/auth/`: การทำเรื่องสมัครสมาชิก ยืนยันโดเมนมหาวิทยาลัย และตั้งรหัสผ่านใหม่
*   `/lib/features/incident/`: การกรอกข้อมูลเหตุด่วน ตารางคะแนนประเมิน รายละเอียดคดีฉุกเฉิน
*   `/lib/features/dispatcher/`: หน้าต่างศูนย์ควบคุมมอนิเตอร์ แผนที่ติดตามพิกัดของศูนย์ประสานงานสั่งการ
*   `/lib/features/responder/`: ระบบทำงานของทีมช่วยเหลือภาคสนาม การปรับเปลี่ยนพิกัดความคืบหน้าการระงับภัย
*   `/lib/features/chat/`: ระบบแชทกลุ่มรายภัยพิบัติ และแชทตรงส่วนบุคคลความปลอดภัย

### ทรัพยากรระบบและสิทธิ์การตั้งค่า (Assets & Config - ลำดับความสำคัญในการพัฒนา: ปานกลาง)
*   `pubspec.yaml`: ไฟล์ติดตั้งระบบปลั๊กอินและระบุโฟลเดอร์มีเดียหลักของโปรเจกต์
*   `firestore.rules` และ `storage.rules`: ไฟล์จำกัดขอบสิทธิ์ความปลอดภัย ป้องกันบัญชีทั่วไปข้ามสิทธิ์แก้ไขข้อมูลสถิติ
*   `/assets/`: รูปโลโก้ของวิทยาเขต และไฟล์ Chimes เสียงบี๊บเตือนภัยไซเรนด่วน (`alert.mp3`)

### ไฟล์ขยะและไฟล์สร้างขึ้นแบบชั่วคราว (Safe to Ignore - ลำดับในการพัฒนา: ปลอดภัยในการมองข้าม)
*   `/android`, `/ios`, `/web`, `/windows`: โฟลเดอร์ที่รวบรวมไฟล์ระบบปฏิบัติการ Native ในจังหวะเตรียมแพ็คแอป
*   `.dart_tool/`, `build/`: รวบรวมชุดคำสั่งและทรัพยากรสร้างขึ้นชั่วคราวในเวลาคอมไพล์โปรแกรม
*   `pubspec.lock`: เอกสารล็อครุ่นเวอร์ชันของไฟล์ปลั๊กอินภายนอก
*   `mobile_user_app.iml`: ไฟล์จำลองโครงการ IDE ซ้ำซ้อนที่สามารถลบออกได้อย่างปลอดภัย โดยคีย์หลักในการใช้งานคือไฟล์ `sut_campus_incident_system.iml`

## 11. คู่มือการเริ่มต้นพัฒนาซอฟต์แวร์ฉบับเข้าใจง่าย (Developer Onboarding Guide)

สำหรับนักพัฒนาหน้าใหม่หรือแอดมินวิทยาเขตที่เข้ามารับช่วงต่อโครงการ เพื่อลดขั้นตอนการเข้าศึกษาซอร์สโค้ด ให้ศึกษาผ่านลำดับขั้นดังต่อไปนี้:

### ขั้นตอนที่ 1: ตรวจเอกสารอ้างอิงเชิงลึก
*   อ่านระบบโครงสร้างและการซิงค์ข้อมูลแชทกลุ่มจากเอกสาร [docs/architecture.md](./docs/architecture.md)
*   ทำความเข้าใจตัวแปรข้อมูล คีย์ฟิลด์ และกฎสิทธิ์ความปลอดภัยในคลังข้อตกลง [docs/database.md](./docs/database.md)

### ขั้นตอนที่ 2: วิเคราะห์สภาพแวดล้อมสเตตข้อมูล
*   เปิดดูไฟล์ `lib/main.dart` เพื่อดูระบบเริ่มต้นของตัวเชื่อมเซิร์ฟเวอร์เลส แผนดักเซสชันหมดอายุ และการเรียกเปิดการทำงานบนเว็บ
*   ศึกษาคู่มือสตรีมข้อมูล Riverpod ในคลาสกลาง `lib/core/providers.dart` เพื่อทำความเข้าใจวิธีการเขียนท่อส่งสัญญาณข้อมูลและการอัปเดตสเตตในฝั่งหน้าจออินเทอร์เฟซ

### ขั้นตอนที่ 3: ตรวจสอบหัวใจการประมวลหลัก
*   สืบค้นตรรกะแผนที่แปลงพิกัดภูมิศาสตร์ในไฟล์ `lib/core/adaptive_map_widget.dart`
*   ดูตรรกะการคัดกรองระดับสเตตคดีความปลอดภัยในไฟล์คลาสสถาปัตยกรรมระดับ Domain `lib/features/incident/domain/responder_logic.dart`

### ขั้นตอนที่ 4: ลำดับขั้นตอนประลองกระบวนการทำงานจริง
*   แกะเส้นทางการแจ้งเหตุ:
    `choose_report_type_screen.dart` $\rightarrow$ `report_incident_screen.dart` $\rightarrow$ `incident_repository.dart`
*   แกะเส้นทางการควบคุมของศูนย์วิทยุประสานงานสั่งการ:
    `dispatcher_screen.dart` $\rightarrow$ `incident_panel.dart` $\rightarrow$ `responders_panel.dart`
*   แกะเส้นทางการช่วยชีวิตกู้ชีพภาคสนาม:
    `responder_dashboard.dart` $\rightarrow$ `navigator_screen.dart`
