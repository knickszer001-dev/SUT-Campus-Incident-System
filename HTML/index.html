📋 สรุปสาระสำคัญของโครงการ (Executive Brief)

โครงการวิจัยและพัฒนานี้เป็นการออกแบบโครงสร้างพื้นฐานดิจิทัลสำหรับความปลอดภัยในสถานศึกษาขนาดใหญ่ โดยเลือกใช้ **มหาวิทยาลัยเทคโนโลยีสุรนารี** เป็นพื้นที่ทดสอบระบบเสมือนจริง โครงสร้างหลักประกอบด้วยโมบายแอปพลิเคชันฝั่งผู้แจ้งเหตุ (นักศึกษา/บุคลากร) หน้าจอนำทางของหน่วยเคลื่อนที่เร็ว (Responder) แผงควบคุมบริหารคดีฝั่งศูนย์วิทยุผ่านเว็บบราวเซอร์ (Dispatcher Panel) เชื่อมต่อด้วยท่อส่งข้อมูลเรียลไทม์ความหน่วงต่ำผ่าน Firebase NoSQL DB และสัญญาณ Push Notification ด่วนพิเศษ

💡 ข้อมูลการปฏิสัมพันธ์: คุณสามารถเข้าถึงข้อมูลเชิงวิชาการและวิศวกรรมซอร์สโค้ดทุกส่วนผ่านเมนูด้านซ้าย หรือใช้ช่องค้นหาด้านบนสุดเพื่อกรองข้อมูลไฟล์หรือฟังก์ชันที่ต้องการตรวจสอบได้ทันที

⚠️ ปัญหาคดีฉุกเฉินในพื้นที่เขตการศึกษาขนาดใหญ่ (Problem Statement)

มหาวิทยาลัยเทคโนโลยีสุรนารีมีพื้นที่รับผิดชอบกว้างใหญ่กว่า **7,000 ไร่** ซึ่งมีภูมิประเทศซับซ้อน ประกอบด้วยอาคารเรียน หอพัก ศูนย์วิจัย ป่าธรรมชาติ และพื้นที่เกษตรกรรม เมื่อเกิดอุบัติเหตุหรือภัยเงียบในสถานศึกษา มักจะประสบปัญหาสำคัญ 3 ประการ:

  • ⚡
    การระบุตำแหน่งคลาดเคลื่อนสูง: ผู้แจ้งเหตุมักไม่ทราบจุดพิกัดหรือชื่ออาคารที่แน่ชัดในเวลากลางคืน ทำให้เจ้าหน้าที่กู้ภัยหลงตำแหน่งเฉลี่ย 10 - 15 นาที
  • ⚡
    การสื่อสารเกิดคอขวด (Information Bottleneck): การแจ้งเหตุทางโทรศัพท์แบบดั้งเดิมรองรับการสนทนาสายต่อสาย ไม่สามารถมองเห็นพิกัด ภาพถ่าย หรืออัปเดตสเตตัสในแบบเรียลไทม์ได้พร้อมกันหลายฝ่าย
  • ⚡
    การจัดสรรงานกู้ภัยไร้ประสิทธิภาพ (Dispatcher Blindspot): ศูนย์ประสานงานไม่ทราบตำแหน่ง Live พิกัดของเจ้าหน้าที่เคลื่อนที่เร็ว ทำให้จัดคดีให้คนไม่ตรงกับจุดพิกัดหรือขาดเครื่องมือพิเศษที่เหมาะสม

🕒 วงจรสเตตัสและการเปลี่ยนผ่านคดีฉุกเฉิน (Incident Lifecycle Timeline)

เพื่อรักษากฎความสมบูรณ์ทางเงื่อนไข ข้อมูลเหตุการณ์จะมีสถานะย่อย 2 เลเยอร์ควบคุมอย่างเข้มงวด:

📌 สถานะหลัก (Core Incident Status)

NEW (เหตุใหม่): ถูกจัดตั้งทันทีเมื่อนักศึกษากดส่ง คดีพร้อมเร่งดึงวิเคราะห์พิกัด
IN_PROGRESS (กำลังประสาน): เจ้าหน้าที่รับเรื่อง ศูนย์ส่งต่อกู้ภัยเคลื่อนกำลังพลลงตรวจพื้นที่
RESOLVED (เสร็จสิ้นคดี): บรรเทาภัยคดีระงับเหตุเรียบร้อย ข้อมูลถูกจัดส่งเข้าคลังล็อกนิติวิทยาศาสตร์
CANCELLED (คดียกเลิก/เท็จ): เหตุจำลอง ผิดสระ หรือยกเลิกเคส ดำเนินการเก็บบันทึกประวัติเพื่อตรวจสอบวินัย

🚑 สถานะกู้ภัยย่อย (Timeline Status Detail)

1. REPORTED (บันทึกเข้าระบบ)

ข้อมูลถูกกระจายเข้า Firebase realtime database และปลุกแผงควบคุมวิทยุการบิน

2. ACCEPTED (กู้ภัยรับคำร้อง)

เจ้าหน้าที่แตะรับเรื่องบนโมบายแอป เสียงไซเรนสงบ ระบบเริ่มคำนวณเวลาการเคลื่อนตัว

3. EN_ROUTE (กำลังเดินทางลงพื้นที่)

ระบบจับตำแหน่ง GPS บนเบื้องหลัง ส่งพิกัดความคลาดเคลื่อนต่ำให้ผู้แจ้งเหตุเฝ้าดูความเร็วรถ

4. ARRIVED (เดินทางถึงจุดปะทะเหตุ)

ดักตรวจตำแหน่งด้วยรัศมี Geofence 20 เมตร แตะปุ่มยืนยัน ปัดหน้าจอเข้าห้องสนทนาเรียลไทม์

5. RESOLVED (ภัยสงบเรียบร้อย)

ปิดการสื่อสาร บันทึกบันทึกข้อมูล พร้อมจัดเก็บข้อมูลลง Audit Logs ครบถ้วน

💡 10 สุดยอดฟีเจอร์นวัตกรรมเพื่อสถาบันการศึกษา (10 Core Features Breakdown)

แพลตฟอร์มของเราไม่ได้สร้างเพื่อแจ้งเหตุธรรมดา แต่รวบรวมฟังก์ชันระดับพรีเมียมไว้ถึง 10 ส่วนสำคัญ:

01

Student Email Bypass Hack

แปลงข้อมูลรหัสนักศึกษาของ มทส. ให้เป็นอีเมลสมมติ เพื่อผ่านด่านคีย์หลักของ Firebase Auth โดยไม่ต้องกังวลเรื่องการกรอกอีเมลยุ่งยากและกรองเบอร์โทรศัพท์ได้แม่นยำ

02

F5 Smart Assignment

ประเมินส่งตัวระงับเหตุฉุกเฉินด้วยค่าน้ำหนักตัวคูณ 3 เลเยอร์: 1) ความเชี่ยวชาญของแผนกตรงภัย 2) ภาระงานกู้ภัยที่ยังว่างอยู่ 3) ระยะห่างทางพิกัดดาวเทียม

03

Case-Isolated Group Chats

ระบบเปิดห้องสนทนาแยกคดีทันทีอัตโนมัติเมื่อเจ้าหน้าที่กู้ภัยรับคดี เพื่อจำกัดพื้นที่การพูดคุยไม่ให้มีข้อมูลปะปนกับคดีอื่น รักษาความลับทางกฎหมาย

04

Adaptive Map Web Bypass

สำหรับศูนย์ประสานงานบนเว็บ จะปรับแผนที่เป็น Leaflet + OpenStreetMap เวกเตอร์หมุดอัตโนมัติ เพื่อเลี่ยงอาการแครชของคอมพิวเตอร์และประหยัดงบค่า API ได้ 100%

05

Live Geofenced Arrival

ตรวจสอบระยะห่างระหว่างเจ้าหน้าที่กับผู้แจ้งเหตุด้วยสัญญาณดาวเทียมมือถือแบบสามเหลี่ยม เมื่อพิกัดเข้าใกล้ต่ำกว่า 20 เมตร จะเปิดปุ่มยืนยันถึงที่เกิดเหตุอัตโนมัติ

06

Audit Trail Subcollection

เก็บรักษาประวัติความเปลี่ยนแปลงคดีพร้อมบันทึก operatorId และการขยับสถานะโดยละเอียดผ่าน `/logs` ซับคอลเลกชันระดับนิติวิทยาศาสตร์ ป้องกันการปิดข้อมูลเท็จ

📄 การทำงานเชิงลึกรายไฟล์ซอร์สโค้ด (File-by-File Analysis)

วิเคราะห์ซอร์สโค้ดของไฟล์สำคัญในระบบเพื่ออธิบายความรับผิดชอบ (File Responsibility) และตรรกะการรัน:

หน้าที่: เป็นจุดเริ่มต้น (Bootstrapper) ในการทำปฏิกิริยากับฮาร์ดแวร์และคลาวด์ก่อนวาดหน้า UI เผยสถาปัตยกรรม Boot flow ดังนี้:

lib/main.dart 
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );
  
  // ตั้งค่า Push Alert และบริการดึงค่าสเปกระยะไกล
  await NotificationService.initialize();
  await RemoteConfigService.initialize();
  
  runApp(
    const ProviderScope(
      child: CampusIncidentApp(),
    ),
  );
}

คำอธิบายเชิงลึก: มีการห่อแอปพลิเคชันทั้งหมดไว้ภายใต้ `ProviderScope` ของ Riverpod เพื่ออำนวยความสะดวกในการใช้ State container เดียวกันตลอดวงจรชีวิตแอป

หน้าที่: จัดการเรื่องการล็อกอิน สมัครสมาชิก และเทคนิค Bypass ระบบของ Firebase เพื่อสนับสนุนระบบระบุสิทธิ์รหัสนักศึกษาของ มทส.:

lib/features/auth/data/auth_repository.dart 
// แปลงรหัสนักศึกษา B6412345 เป็นอีเมลจำลองเพื่อให้ทำงานร่วมกับ Firebase Auth ได้
final studentEmail = '$normalizedId@campus.local';

UserCredential result = await _auth.signInWithEmailAndPassword(
  email: studentEmail,
  password: password,
);

คำอธิบายเชิงลึก: เนื่องจาก Firebase Auth บังคับใช้อีเมลในการทำระบบ Authentication แต่สำหรับแอปของมหาวิทยาลัยเราต้องการให้ผู้ใช้ใช้เพียงรหัสนักศึกษา เราจึงใช้วิธีแปลงรหัสเป็นอีเมลสมมติ `รหัส@campus.local` เพื่อรองรับกระบวนการเก็บรักษาความปลอดภัยของ GCP ไร้กังวล

หน้าที่: นำทางผู้ใช้ที่ล็อกอินเสร็จเรียบร้อย ไปที่หน้า UI ตามบทบาท (Role-based view):

lib/features/auth/presentation/role_redirect.dart

final role = doc.data()?['role'] ?? 'user';

switch (role) {
  case 'dispatcher': return DispatcherScreen();
  case 'responder':  return ResponderDashboard();
  case 'admin':      return AdminDashboard();
  default:           return HomeScreen(); // สิทธิ์นักศึกษาทั่วไป
}

คำอธิบายเชิงลึก: ในหน้าต่างตรวจสอบสิทธิ์การนำทางจะเรียกใช้ `ensureUserProfile` ดักจับกรณีที่ผู้ใช้ล็อกอินสำเร็จแต่ฐานข้อมูล Firestore ยังไม่ได้ทำโครงสร้าง หรือไม่มีข้อมูลโปรไฟล์ เพื่อความสมบูรณ์และไร้ปัญหาหน้าจอแครช

หน้าที่: ศูนย์บัญชาการรหัสการเข้าถึง Firestore สำหรับสร้างคดี แจ้งล่วงหน้า อัปเดตสเตตัสพร้อมเขียนล็อก Audit Log อัตโนมัติ ป้องกันข้อมูลชนกันโดยใช้ Transaction:

lib/features/incident/data/incident_repository.dart 
Future<bool> updateIncidentStatus(String incidentId, String newStatus) async {
  final docRef = _firestore.collection('incidents').doc(incidentId);
  
  return await _firestore.runTransaction((transaction) async {
    final snapshot = await transaction.get(docRef);
    final currentStatus = snapshot.data()?['status'];
    
    // ตรวจสอบ State transition เพื่อรักษาความสมบูรณ์ทางเงื่อนไข
    if (!ResponderLogic.canTransition(currentStatus, newStatus)) {
      throw Exception("Invalid status change sequence!");
    }
    
    transaction.update(docRef, { 'status': newStatus });
    return true;
  });
}

คำอธิบายเชิงลึก: นอกเหนือจากการใช้ Transaction ในการควบคุมความสมบูรณ์ในการอัปเดตสถานะแล้ว ทุกครั้งที่บันทึกข้อมูลเรียบร้อย จะมีการสั่งบันทึกการกระทำลงในซับคอลเลกชัน `/logs` ทันที เพื่อบันทึกประวัติการกระทำของยูสเซอร์ในคดีนั้น ๆ แบบอ้างอิงนิติวิทยาศาสตร์

📱 แผนผังและลอจิกแต่ละหน้าจอการทำงาน (Screen-by-Screen Flow)

แอปพลิเคชันของเรามีระบบความปลอดภัยและส่วนประสานการทำงานสลับไปมาตามหน้าจอหลักถึง 4 หน้าจอ เพื่อการตอบสนองที่แยกความรับผิดชอบอย่างชัดเจน:

หน้าจอการใช้งาน (Core UI Screen) บทบาทหลักที่มองเห็น (Target Role) กลไกและการปฏิสัมพันธ์ข้อมูลเบื้องหลัง (Backend Interaction Flow) การทำงานกับ Firestore (Database Reads/Writes)
Login & Profile screen ผู้ใช้ทุกคน (All Users) รับข้อมูลรหัสแปลงเป็นอีเมลสมมติ เช็คสิทธิ์ ล็อกอิน สำเร็จสลับเข้าหน้าต่างสิทธิ์ Read: ตรวจเอกสารใน `/users/{uid}` สังกัดสิทธิ์บทบาท
SOS Incident Report flow นักศึกษา/บุคลากร (User) เลือกประเภทภัย กรอกรายละเอียด บังคับดึงตำแหน่ง GPS แนบรูปภาพหน้างาน ยื่นเรื่อง SOS Write: บันทึกเอกสารเหตุการณ์ใหม่ลง `/incidents`
Web Dispatcher Control Panel ศูนย์สั่งการ (Dispatcher) แสดงแผนที่ Live Map ดักฟังคดีเข้าใหม่ด้วยสัญญาณไซเรน อัลกอริทึมแนะนำ Responder กดเลือกคนกู้ภัย Read & Write: snapshots listeners incident, updates status และ assignedResponder
Responder Mobile Dashboard กู้ภัย (Responder) รับ Notification แจ้งเตือนด่วน สั่นและเสียงไซเรน กดดูรายละเอียดคดี แตะนำทาง Google Maps แชทเฉพาะเคส กดอัปเดต Timeline การเดินทาง Write: snapshots location details ใน `/users/{uid}`, และ updates status timeline ใน `/incidents/{incidentId}`

🗄️ โครงสร้างฐานข้อมูล Cloud Firestore NoSQL (Database Schema)

ออกแบบเป็นแบบเอกสารแบนราบ (Flat Document) เพื่อลดจังหวะทราฟฟิก และแยกเก็บข้อมูลประวัติที่เกิดการปฏิสัมพันธ์บ่อยเป็น Subcollection:

เก็บข้อมูลคุณลักษณะของผู้ใช้งานและพิกัด Live ล่าสุด สำหรับ Responder ในระบบ:

📍 ระบบระบุตำแหน่งพิกัดดาวเทียมและแผนที่อัจฉริยะ (GPS Geolocation & Mapping)

การระบุสถานที่ในเขตพื้นที่มากกว่า 7,000 ไร่ ของ มทส. มักจะหลงพิกัดบ่อยครั้ง ระบบของเราแก้ปัญหานี้เชิงวิศวกรรมโดยการประสานงาน Native GPS ของฮาร์ดแวร์มือถือ ร่วมกับระบบสลับแผนที่อัตโนมัติ (Adaptive Mapping Engine) ดึงตำแหน่งพิกัดและคำนวณระยะนำทางดังนี้:

ใช้แพ็กเกจ geolocator ดึงตำแหน่งทางภูมิศาสตร์จากเสารับสัญญาณโทรศัพท์เคลื่อนที่แบบสามเหลี่ยมและจีพีเอสดาวเทียมโดยตรง โดยระบบแอปมือถือถูกกำหนดค่าความแม่นยำสูงสุด LocationAccuracy.high สำหรับดึงพิกัดแจ้งเหตุ และ LocationAccuracy.bestForNavigation สำหรับฝั่งเจ้าหน้าที่นำทางเพื่อลดความผิดพลาดพิกัดให้ต่ำกว่า 10 เมตร โดยเก็บพิกัดในรูปแบบฟิลด์ NoSQL GeoPoint ประกอบด้วย [Latitude, Longitude]

สำหรับการจัดสรรสิทธิ Smart Assignment ของศูนย์ประสานงาน ระบบจะนำพิกัดจุดเกิดเหตุไปสืบค้นหาเจ้าหน้าที่กู้ภัยที่อยู่ใกล้เคียงที่สุดโดยนำทางพิกัดดาวเทียมมาคำนวณผ่าน Haversine Formula ซึ่งช่วยประเมินความยาวส่วนโค้งบนพื้นผิวทรงกลมของโลกโดยไร้การพึ่งพา API ภายนอกเพื่อหลีกเลี่ยง Latency ดังนี้:

Haversine Calculation Code 
double calculateDistance(double lat1, double lon1, double lat2, double lon2) {
  const double p = 0.017453292519943295; // Pi / 180 (rad)
  var a = 0.5 - cos((lat2 - lat1) * p)/2 + 
          cos(lat1 * p) * cos(lat2 * p) * 
          (1 - cos((lon2 - lon1) * p))/2;
  return 12742 * asin(sqrt(a)); // 2 * R; R = 6371 km
}

🛡️ กฎความปลอดภัยเครือข่ายและการป้องกัน Firebase (Security Rules & Auth Gate)

เนื่องจากฐานข้อมูล Firestore และ Storage เก็บข้อมูลอ่อนไหวระดับบุคคล เราจึงควบคุมระบบด้วยกฎ **Firestore Security Rules** ที่มีการยืนยันตัวตนระดับ Token ดังนี้:

firestore.rules 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // ดักผู้ใช้ทุกคนต้องล็อกอินก่อนเสมอ
    function isAuthenticated() {
      return request.auth != null;
    }
    
    // ดักกรองบทบาทของผู้ใช้งานจากโปรไฟล์
    function getUserRole() {
      return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role;
    }
    
    match /users/{userId} {
      allow read: if isAuthenticated();
      allow write: if isAuthenticated() && (request.auth.uid == userId || getUserRole() == 'admin');
    }
    
    match /incidents/{incidentId} {
      allow read: if isAuthenticated();
      allow create: if isAuthenticated();
      allow update: if isAuthenticated() && (getUserRole() in ['dispatcher', 'responder', 'admin']);
      
      match /messages/{messageId} {
        allow read, write: if isAuthenticated();
      }
      
      match /logs/{logId} {
        allow read: if isAuthenticated();
        allow write: if false; // เขียนประวัติ Audit Log ผ่าน Cloud Function เท่านั้น!
      }
    }
  }
}

กลไก Auth Gate ความปลอดภัย: สังเกตว่าในคอลเลกชัน `/logs` สิทธิ์ในการเขียนถูกกำหนดให้เป็น `false` เสมอ เพื่อห้ามไม่ให้แอปฝั่งเครื่องมือถือเขียนทับประวัติล็อกช่วยคดีฉุกเฉินได้เอง ซึ่งเป็นหลักปฏิบัติที่ช่วยป้องกันการทำลายหลักฐานหรือทุจริตความรับผิดชอบได้อย่างเด็ดขาด

🧠 Domain Layer — รายละเอียดกฎตรรกะหัวใจแอป

ชั้น Domain ถือเป็น "มันสมอง" ของระบบ มีความเป็นอิสระสูงสุด (Pure Layer) โดยไม่มีการ Import แพ็กเกจหรือคีย์ภายนอกใด ๆ เข้ามาเกี่ยวข้อง:

  • Incident Entity Model: ก้อนโครงสร้างวัตถุข้อมูลหลัก เช่น ประเภทภัย พิกัดละติจูด-ลองจิจูด และรายชื่อเอกสารรูปภาพ
  • Responder Logic (State Machine): คุมกฎความถูกต้อง เช่น ห้ามกดปิดคดี (RESOLVED) หากสถานะปัจจุบันยังเป็นเหตุแจ้งใหม่ (NEW)
  • Haversine Formula Interface: กำหนดสัญญาในการคำนวณพิกัดระยะทาง แนะนำ Responder ที่มีรัศมีและสัญชาติงานเหมาะสมที่สุด
